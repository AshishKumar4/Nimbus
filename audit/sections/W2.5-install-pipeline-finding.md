# W2.5 Finding — Install-Pipeline Tarball Extraction Gap

> **Status:** Uncovered during W2 prod verification (2026-04-30, prod
> ver `22962f4d`). NOT a W2 regression — pre-existing but masked by the
> broken pre-W2 resolver. **Dispatch separately as W2.5 before W3.**

## Symptom

`npm install <pkg>` reports successful install ("added N packages, M
files") but several transitive packages have empty or partially-empty
`node_modules/<pkg>/` directories. Verified via `fs.readdirSync` on
fresh prod sessions:

```
npm install fastify   → reports "added 46 packages (1928 files)"
                      → fs.readdirSync(node_modules/avvio) === []
                      → fs.readdirSync(node_modules/fastq) === []
                      → fs.readdirSync(node_modules/pino)  === []
                      → fs.readdirSync(node_modules/semver) === []
                      → fs.readdirSync(node_modules/@fastify/error) === [.gitattributes, ...18 files]  (works)
```

Same pattern verified for: express (`get-intrinsic`, `es-object-atoms`),
ts-jest (`typescript`), drizzle-orm (`pg-core/columns/`), redis
(`@redis/client/dist/`), puppeteer-core (`lib/cjs/puppeteer/api/`),
framer-motion + react@18.3.1-pinned (`react/cjs/`).

## Why W2 unmasked it

The pre-W2 runtime resolver (`src/node-shims.ts:889-913 __resolvePkgEntry`)
was a broken hand-roll that only honoured top-level
`require|default|import` of the exports field. For express:

  - **Pre-W2 error**: `Cannot find module './lib/express' (from .../express)`
    (resolver tried main → got `'index.js'` which `require()`s `'./lib/express'`,
    which the resolver couldn't probe properly because of the relative-path
    fromDir corruption bug)
  - **Post-W2 error**: `Cannot find module 'es-object-atoms' (from
    .../get-intrinsic)` (resolver successfully loaded
    `express/index.js → ./lib/express.js → ... → get-intrinsic`, then
    tried `require('es-object-atoms')` which the resolver correctly
    walked node_modules to find — but the directory was empty)

So pre-W2's failure was always the *first* thing the broken resolver
tripped on. Post-W2 the resolver is correct, so execution advances
deeper and hits the install gap.

## Affected probe-set packages (measured)

8 of 29 post-W2 ⚠️ packages are blocked by this gap. Each verified by
direct `fs.readdirSync` on prod:

| Package          | Empty directory                                              |
|---|---|
| `express`         | `get-intrinsic/`, `es-object-atoms/`                         |
| `fastify`         | `avvio/`, `fastq/`, `pino/`, `semver/`                      |
| `ts-jest`         | `typescript/`                                                |
| `drizzle-orm`     | `drizzle-orm/pg-core/columns/`                              |
| `redis`           | `@redis/client/dist/` (entire dist subdirectory)            |
| `puppeteer-core`  | `puppeteer-core/lib/cjs/puppeteer/api/`                     |
| `radix-react-dialog` | `@radix-ui/react-use-layout-effect/`                     |
| `framer-motion`   | `react/cjs/` (only when react@18.3.1 is pinned)             |

Unblocking all 8 would land **W2's audit-target threshold (12-14 ✅)**
without any other wave work.

## Hypotheses (in order of plausibility)

### H1 — `streamTarEntries` race in `src/parallel/generated-workers.ts`

Most likely. The batch-facet runs multiple tarballs through the same
extraction pipeline at internal `pLimit=3`. If two tarballs target the
same hoisted directory at near-identical timestamps, the second tarball
may overwrite or interrupt the first.

Evidence supporting:
- Different transitive packages from the *same* install command get
  different treatment (avvio empty, @fastify/error full).
- `react@18.3.1` (separate install command) leaves `react/cjs/` empty;
  bare `react` (latest) populates it. Suggests version-specific tarball
  shape interacts with extraction order.

Test: install fastify with `pLimit=1` and check whether avvio populates
correctly. If yes → H1 confirmed.

### H2 — `parseTarHeader` USTAR-extension parsing

Possible. The `src/parallel/generated-workers.ts:parseTarHeader` may
fail silently on USTAR-extended entries (long file names, sparse files,
extended attribute headers) used by newer npm packages. A failed parse
that's swallowed silently would skip the rest of the entry stream.

Evidence supporting:
- Pattern is consistent: certain packages (avvio, fastq, pino, semver)
  always come up empty across multiple sessions; others always populate.

Test: log every parseTarHeader return and compare entry count vs
upstream npm tarball entry count.

### H3 — Hoist-target overwrite in `npm-installer.ts`

Lower-likelihood but possible. The dedupe/hoist logic in
`resolvePackageEntryPath` may resolve two different tarballs to the
same target directory and the second arrival could clobber.

### H4 — VFS write batching dropping pending entries

The batch-facet writes to VFS in batches. A batch boundary that lands
mid-tarball could lose entries if the next batch overwrites
incorrectly. Look at `batchWriteRows` vs total-entry counts in
`/api/stats` after a known-failing install.

## Investigation plan (suggested for W2.5 dispatch)

1. **Probe install with file count**: instrument batch-facet to log per-tarball
   entry-count vs file-count-at-VFS-write-time. Diff identifies dropped entries.
2. **pLimit=1 reproduction**: run `npm install fastify` with batch-facet
   internal pLimit forced to 1. If avvio populates → H1.
3. **Tarball entry log**: log `parseTarHeader` return for first 50 entries
   of an avvio tarball. Compare against `tar -tvf` of the upstream tarball.
   If counts diverge → H2.
4. **VFS readback verify**: after install, sha-256 every file written and
   compare to upstream tarball. Mismatches → write-corruption (H4).

Estimated effort: **S (3-5 days)** to investigate + S (1-3 days) to fix
once root cause is known.

## Estimated impact

If install-pipeline gap is fixed:
- **+8 ✅** on the 33-package probe set (target hit: 12 ✅ matches the
  audit's "12-18" W2 target)
- **+~15-25%** on full npm coverage (transitive deps form a larger share
  than direct deps in real-world projects)
- **Mossaic regression** — likely unaffected (Mossaic happens to use packages
  that don't trigger the gap)
