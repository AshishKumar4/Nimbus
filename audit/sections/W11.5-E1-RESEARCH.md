## TL;DR

Three high-impact findings invalidate parts of W8-retro and W11.5-retro:

1. **`enable_nodejs_v8_module` is default-on at compatibility date `2026-03-17`** — Nimbus is at `2026-04-01`, so `node:v8` is exposed today. **But:** workerd's `v8.serialize` / `v8.deserialize` are stub-throws (`ERR_METHOD_NOT_IMPLEMENTED`) per the upstream source. The flag enables the *module shape*, not the binary serializer. So we cannot route IPC through `node:v8`.
2. **`structuredClone()` global has been in Workers since 2021-11-19.** It is the V8-serializer-equivalent for **the union of types jest-worker actually sends** (Buffer, Date, Map, Set, TypedArray, plain objects, errors with the `enhanced_error_serialization` flag — default on at compat date `2026-04-21`). It does NOT support `Function`, RpcStub, or transfer of arbitrary host objects, but jest-worker doesn't send any of those.
3. **`MessageChannel` / `MessagePort` are available** (since `2025-08-15`), but the workerd implementation is **same-isolate only** — postMessage clones values, never crosses RPC boundaries (workerd `messagechannel.h` line 17: *"the MessagePort only ever passes messages around within the same isolate"*). For cross-facet IPC we still need explicit RPC.

**Recommendation:** Pick **B1 (full LOADER.get child + bidirectional WebSocket-as-IPC)** but upgrade the **frame format** from JSON-newline to `structuredClone-via-WebSocket-binary-frame`. That's the smallest viable substrate change to lift Nimbus's W8 fork() from "JSON projection" to "structured-clone-equivalent" — enough to unblock jest-worker + Next.js's worker pool entry path. The WebSocket route is preferred over the existing cpStdinWrite/cpReadStdin RPC route because (a) WebSocket frames are binary-native (no string-encoding round-trip), (b) the 32 MiB-per-frame limit (since `2025-10-31`) matches RPC structured-clone, (c) hibernation already works (W9). Cost: ~3-5 days; one new RPC method (`cpForkChannel(childPid)` returning a `WebSocketPair`); shim swap of `child.send` from JSON-stringify to `port.postMessage`.

**Decisions needed from user before build dispatch** (see §F): scope of structured-clone fidelity (is "Buffer + Date + Map + Set + TypedArray" enough, or do we need full structured-clone including SharedArrayBuffer? — answer: SAB is forbidden anyway, so "Buffer + Date + Map + Set + TypedArray" is the bar); B5 RFC-to-CF tradeoff (file an issue at workerd discussions for native fork primitive, but don't block on it); and whether to ship E1 *before* E2 (recommended — it's the smaller substrate; E2 builds on top).

# §A. Workerd's IPC primitive landscape

What primitives does workerd give us for getting bytes from one isolate to another? This section enumerates each and verdicts it for "could it serve as fork() IPC?"

## A.1 `ctx.exports` loopback (used today)

**Status:** ✅ Default-on at compat date `2025-11-17` (Nimbus `2026-04-01` ≥ default).
**Source:** [Cloudflare changelog 2025-09-26: Automatic loopback bindings via ctx.exports](https://developers.cloudflare.com/changelog/post/2025-09-26-ctx-exports/); [`enable_ctx_exports` flag docs](https://developers.cloudflare.com/workers/configuration/compatibility-flags/#enable-ctxexports).

`ctx.exports.<ClassName>` returns a Service Binding whose backing class is the same Worker's top-level export. This is the substrate that lets [`src/supervisor-rpc.ts`](../../src/supervisor-rpc.ts) be reachable from facet code via `env.SUPERVISOR.cpStdinWrite(...)` etc. — the loopback pattern Nimbus's W8 already uses.

**Cross-isolate?** Yes. The facet runs in its own dynamic-worker isolate (LOADER); the supervisor-RPC class runs in the parent worker's isolate. RPC crosses the isolate boundary.

**Frame format?** RPC values use JS structured-clone for arguments + return values. Cap is 32 MiB per call ([RPC docs §Limitations](https://developers.cloudflare.com/workers/runtime-apis/rpc/)). Functions, RpcStub, ReadableStream pass via the capability system.

**As fork IPC?** Already used today (W8). Frame format is fine for the wrapper but the *user message payload* gets JSON-stringified by the W8 shim before crossing — that's the lossy step. Lever: **lift the user payload from JSON to structuredClone-shape passed as an RPC argument directly**. Could work but RPC adds round-trip latency (every send is one RPC call).

**Verdict:** ✅ Foundational primitive but not optimal as the on-the-wire IPC frame format. Use for control plane (cpForkChannel handshake) not data plane (per-message send).

## A.2 `LOADER.get()` dynamic-worker spawn (used today)

**Status:** ✅ GA. Nimbus W8 uses it for facet-direct execution.
**Source:** [Dynamic Workers API reference](https://developers.cloudflare.com/dynamic-workers/api-reference/); used at [`src/facet-process.ts`](../../src/facet-process.ts) lines 1-50 (header).

`env.LOADER.get(id, factoryFn)` mints (or reuses) a dynamic worker with a fresh isolate. The factoryFn returns `{ env, modules, ... }` which includes Service Bindings — including loopback bindings via `ctx.exports`. This is how a facet receives `env.SUPERVISOR`.

**Cold-start cost:** ~30-100ms typical, varies by code size. Per W8-retro §3 item 1: "we mint a fresh isolate" per facet rather than copy-on-write fork.

**As fork IPC:** the *spawn* mechanism — answers the "how do we get an isolate" question, not the "how do we exchange messages" question.

**Verdict:** ✅ Use as-is. E1 doesn't change spawn semantics.

## A.3 `MessageChannel` / `MessagePort` (same-isolate only)

**Status:** ✅ GA at compat date `2025-08-15`. Nimbus has it.
**Source:** [Cloudflare changelog 2025-08-11: MessageChannel and MessagePort](https://developers.cloudflare.com/changelog/post/2025-08-11-messagechannel/); [MessageChannel API docs](https://developers.cloudflare.com/workers/runtime-apis/messagechannel/).

The workerd implementation (verified by fetching [`src/workerd/api/messagechannel.h`](https://github.com/cloudflare/workerd/blob/main/src/workerd/api/messagechannel.h)) is intentionally restricted:

> *"the MessagePort only ever passes messages around within the same isolate (that is, we're not sending the serialized data off anywhere, we're just cloning it and dispatching it.)"*
>
> *"Transfer lists are currently not supported. This means that you will not be able to transfer ownership of objects like ArrayBuffer or MessagePort between ports."*
>
> *"The MessagePort is not yet serializable. This means that you cannot send a MessagePort object through the postMessage method or via JSRPC calls."*

There IS a TODO comment in the source (line ~80) reading: *"TODO(soon): Support serialization/deserialization to use MessagePort with JSRPC. We'll need to implement a rpc mechanism for passing the messages across the rpc boundary."*

**As fork IPC:** No. MessagePort cannot cross the isolate boundary. **It WOULD work if Nimbus emulated fork() inside a single isolate** (B3), but B3 has its own problems (no real concurrency, see §B.3).

**Verdict:** ⚠️ Useful **inside** an isolate (e.g. for a hypothetical jest-worker shim that pretends to fork but actually runs synchronously in-isolate via MessagePort). NOT useful for cross-facet IPC today. Track the workerd TODO; if it ships, MessagePort becomes a candidate for the post-W8 frame format too.

## A.4 WebSocket-as-IPC (used today between supervisor and HMR)

**Status:** ✅ GA. Nimbus uses it for cirrus-hmr (browser ↔ supervisor) and for `/api/logs/<pid>` (browser ↔ supervisor process logs).
**Source:** [`src/process-logs-api.ts`](../../src/process-logs-api.ts) lines 11-21 (W9 hibernatable); [`src/nimbus-session.ts:1465`](../../src/nimbus-session.ts) shows `acceptWebSocket(server, ['cirrus-hmr'])`; [Durable Objects WebSocket Hibernation API](https://developers.cloudflare.com/durable-objects/best-practices/websockets/).

WebSockets are full-duplex byte/text channels with frame-level semantics. The 1-MiB frame limit was raised to **32 MiB** on `2025-10-31` per [Cloudflare changelog 2025-10-31: Workers WebSocket message size limit increased from 1 MiB to 32 MiB](https://developers.cloudflare.com/changelog/post/2025-10-31-increased-websocket-message-size-limit/). Same as RPC structured-clone.

**Frame format:** application chooses. Today Nimbus sends JSON over text frames for cirrus-hmr, JSON over text frames for process logs. Could send `structuredClone`-shape over **binary frames** by using the global `structuredClone()` to clone a value, then a custom wire format (e.g. CBOR or `JSON.stringify` with a schema) to byte-encode the cloned tree.

**Cross-isolate?** Yes. WebSockets are byte-level protocol; both endpoints can be in any isolate (or in browsers). The supervisor's `webSocketMessage` handler receives bytes/text; the facet end can be a `WebSocketPair` peer obtained via `new WebSocketPair()` and accept-then-passed-through-RPC.

**Hibernation?** Yes, with `ctx.acceptWebSocket(server, tags)` ([Durable Objects WebSocket Hibernation API](https://developers.cloudflare.com/durable-objects/best-practices/websockets/)). Already used by W9.

**As fork IPC:** ✅ Strong candidate. The pattern: supervisor mints `WebSocketPair`, retains one half as the parent end, sends the other half *through RPC* into the dynamic-worker facet during spawn (RPC supports passing WebSocket per the Workers RPC class-instances surface). Then both ends communicate frame-by-frame **without** further RPC roundtrips.

⚠ **Caveat**: passing a `WebSocket` through RPC into a dynamic-worker LOADER child needs verification. The RPC docs say `Request`/`Response`/`ReadableStream`/`WritableStream` cross. WebSocket itself is technically a host object; the RPC class-instances doc says you can pass *RpcTarget instances* across, plus a few specific Web standard types. **This is the highest-value verification probe** before E1 build.

**Verdict:** ✅✅ Best primitive for fork() IPC IF the WebSocket-through-RPC handoff works. If it doesn't, fall back to A.1 (RPC-call-per-message).

## A.5 Structured-clone over RPC (32 MiB wall, W7 streams bypass)

**Status:** ✅ GA. Used by every RPC call today.
**Source:** [Workers RPC docs §Limitations](https://developers.cloudflare.com/workers/runtime-apis/rpc/) — *"The maximum serialized RPC limit is 32 MiB. Consider using ReadableStream when returning more data."*

RPC arguments and return values flow through the [structured-clone algorithm](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API/Structured_clone_algorithm). This is the same algorithm as `structuredClone()`, MessagePort.postMessage, postMessage to Web Workers, and IndexedDB serialization. So the RPC arg type set IS exactly the structured-clone set.

**For fork IPC payloads from jest-worker:** structured-clone supports Buffer (as Uint8Array on the receive side, but workerd auto-rebrands TypedArray subtypes), Date, Map, Set, RegExp, ArrayBuffer, TypedArray, plain objects, recursive references, Errors (with `enhanced_error_serialization` flag — default on at compat date `2026-04-21` per [the flag docs](https://developers.cloudflare.com/workers/configuration/compatibility-flags/)).

**32 MiB cap:** sufficient for jest-worker. jest-worker messages are option-blob-with-args structures; even a 10k-line source file in `setupArgs` is < 1 MiB. Hard cap reached only by truly enormous transfers (raw build artifacts), which are out of jest-worker's normal request shape.

**Verdict:** ✅ Frame format is correct. The question is *which transport* to use: RPC-arg per-message (A.1) vs WebSocket binary frame (A.4) vs MessagePort (A.3, blocked by same-isolate restriction).

## A.6 Shared memory possibilities (SAB, transferList)

**Status:** ❌ Forbidden in workerd by design.
**Source:** workerd has no `SharedArrayBuffer` or `Atomics` exposed at user scope (see Workers limits + the `Workers WebAssembly` doc which notes *"Threading is not possible in Workers. Each Worker runs in a single thread, and the Web Worker API is not supported"* per [Workers WebAssembly docs](https://developers.cloudflare.com/workers/runtime-apis/webassembly/)).

Even if `SharedArrayBuffer` existed, isolates do not share memory — each isolate is a separate V8 heap. SAB is per-V8-isolate.

**Verdict:** ❌ Off-limits. We will not have shared memory between facets. Any IPC must be copy-by-value (which is what V8.serialize and structured-clone provide anyway).

**Key takeaway:** workerd has filled in the *module shapes* for all the Node IPC-related modules but **not the binary serializer or the cross-isolate transport for shared-memory primitives**. For Nimbus's IPC frame format, the available primitives are:

| Available primitive | Cross-isolate? | Frame format | Verdict |
|---|---|---|---|
| RPC method call | ✅ | structured-clone | Round-trip per call |
| `ctx.exports` loopback | ✅ | structured-clone | Same as RPC |
| WebSocketPair via RPC | ✅ (if WS-through-RPC works) | bytes — choose your own | Best fit for streaming IPC |
| `MessagePort` postMessage | ❌ same-isolate only | structured-clone | Blocked for cross-facet |
| TCP / sockets | ❌ | n/a | Off-limits |
| Shared memory | ❌ | n/a | Off-limits |
| `node:v8.serialize` | n/a (stub) | n/a | Wait for upstream |

So the IPC channel for E1 is:

```
spawn:                  LOADER.get + ctx.exports loopback (existing)
control plane:          RPC methods (cpForkChannel, cpKill, cpWait — existing)
data plane (E1):        WebSocketPair piped through RPC, structured-clone frames
fallback data plane:    RPC method per send (slower; still works)
```

## A.8 Wire-format vs transport matrix

Cross-tabulating the three orthogonal axes (frame format, transport mechanism, isolation):

| Frame format | Transport | Cross-isolate? | Available today? | Use case |
|---|---|---|---|---|
| JSON-newline (text) | stdin/stdout queue + cpStdinWrite/cpReadStdin RPC | ✅ via cpStdin* RPC | ✅ W8 today | Legacy fork() — JSON-only payloads |
| `structuredClone()` JS values | RPC argument (auto-cloned) | ✅ | ✅ E1 picks this | jest-worker, vitest, mocha — full Buffer/Date/Map fidelity |
| Bytes (binary frame) | WebSocketPair (if WS-via-RPC handoff works) | ✅ if probe A0 passes | ⚠ unverified | Future high-throughput streaming IPC |
| `structuredClone()` JS values | MessagePort `postMessage` (same isolate) | ❌ same-isolate only | ✅ but useless for fork | Future when workerd lifts cross-RPC restriction |

| Shared memory | SharedArrayBuffer + Atomics | ❌ Workers single-thread | ❌ off-limits | n/a |

**Pick:** structured-clone-over-RPC-arg (row 2). All platform substrates needed are already in production.

## A.9 Performance characteristics — RPC vs alternatives

Approximate latencies, sourced from Nimbus's existing W8 benchmarking (`audit/probes/w8/regression/` runs) plus public CF docs:

| Primitive | Per-call latency (in-process loopback) | Throughput (calls/sec) | Cap |
|---|---|---|---|
| ctx.exports RPC method (no I/O) | ~0.5-2 ms | ~5000/sec/stub | 1000 subreq/req paid plan default; 10k bumpable |
| LOADER.get cold-start | 30-100 ms | n/a (one-shot per spawn) | n/a |
| LOADER.get warm reuse | <1 ms | n/a | n/a (warm slot held by RPC stub) |
| WebSocket frame (same-DO) | <1 ms | high (bounded by DO CPU) | 32 MiB per frame |
| MessagePort postMessage (same-isolate) | <0.1 ms | very high | structured-clone size cap (no doc'd number; bounded by isolate heap = 128 MiB) |
| structuredClone() global call | ~0.05 ms for small objects | n/a | n/a |
| `JSON.stringify` (plain object) | ~0.1 ms | n/a | n/a |
| `JSON.stringify` (Buffer-bearing) | ~0.5 ms (lossy projection) | n/a | n/a |

**For jest-worker INITIALIZE round-trip:** structured-clone-over-RPC adds ~1-3 ms vs JSON-over-RPC (~0.5-1 ms). Acceptable.

**For per-task transform calls:** ~1-3 ms RPC + actual terser work (~10-200 ms) + ~1-3 ms response RPC. RPC overhead is <5% of total.

**For HMR:** ~10-100 messages per HMR cycle. ~30-300 ms RPC overhead per cycle. Imperceptible to user (HMR target is <500 ms total).

Net: RPC overhead is bounded and small. **No performance concern for E1.**

# §B. fork() emulation strategies

For each strategy: viability, prototype probe sketch, predicted unlock surface.

## B.1 Spawn full LOADER.get child + bidirectional WebSocket IPC (W8 pattern, scaled up)

**Strategy:** use the existing W8 facet-as-process spawn substrate. Replace the JSON-newline-over-stdin IPC with a **`WebSocketPair`-piped-through-RPC** channel that carries `structuredClone`-shape frames.

**Substrate change required:**
- Supervisor: new RPC method on `SupervisorRPC`: `cpForkChannel(childPid: number)` → returns the parent half of a `WebSocketPair`. The supervisor retains the child half and stores it on the [`src/facet-process.ts ChildEntry`](../../src/facet-process.ts) record.
- Facet (child): on first `process.send` or `process.on('message')`, the runtime calls `env.SUPERVISOR.cpForkChannel(myPid)` to obtain its WebSocket end. From then on, `child.send(msg)` is `port.postMessage(msg)` and `process.on('message', cb)` is `port.addEventListener('message', cb)`.

**Wire format:** application-defined. Two viable shapes:
1. **Direct `port.postMessage(structuredClone-shape)`** — IF MessagePort cross-isolate gets fixed (workerd TODO). Today blocked.
2. **`port.send(JSON.stringify(serializeForRPC(msg)))` over text frames** — JSON with a Buffer/Date/Map shim layer. Same projection W8 uses today, just on a different transport. Lossy.
3. **Custom binary encoding** — implement a JS-side CBOR or msgpack encoder. Not lossy. ~150 LOC. **Recommended.**

⚠ **The actually best shape:** convert structured-clone to binary via the **`Response`/`Request` body** trick. RPC docs say `Response` instances flow over RPC ([RPC streams docs](https://developers.cloudflare.com/workers/runtime-apis/rpc/)). A `Response` body can be any structured-cloneable value via `Response.json(value)` (uses JSON, lossy) OR via `new Response(new Blob([cborEncode(value)]))` for binary CBOR. So the wire shape is:

```ts
// audit-only sketch, NOT for src/ — pseudo-code for the build wave
class SupervisorRPC {
  async cpForkChannel(childPid: number): Promise<WritableStream<Uint8Array>> {
    const child = this.children.get(childPid);
    const { readable, writable } = new TransformStream();
    child.parentToChildStream = writable;       // server retains
    child.parentToChildReader = readable.getReader();
    return /* the writable end is sent over RPC into the child */;
  }
}
```

Actually simpler: pass a **`WebSocketPair`** into the child via `env.LOADER.get()`'s `env` field. The CF docs say `env` is structured-clone-able + can carry Service Bindings ([Dynamic Workers API reference](https://developers.cloudflare.com/dynamic-workers/api-reference/)). WebSocket is a host object — verify it crosses; if not, use the Service-Binding pattern (loopback `ctx.exports.IpcChannel({ props: { childPid } })` returning a class with `postMessage` / `onMessage`).

**Prototype probe sketch (~10 lines, write before build):**
```js
// audit/probes/w115-e1-investigation/A1-ws-through-loader-env.mjs
// Verify that env.LOADER.get can pass a WebSocket end into a dynamic worker.
// Run against local wrangler-dev:
const { 0: client, 1: server } = new WebSocketPair();
client.accept(); server.accept();
const child = env.LOADER.get('test', () => ({
  modules: { 'main.js': 'export default { fetch(req, env) { return new Response("ok-" + (env.PORT ? "got" : "missing")); } }' },
  mainModule: 'main.js',
  env: { PORT: client },
}));
const r = await child.getEntrypoint().fetch(new Request('http://x/'));
console.log(await r.text());
// Expected: "ok-got" if WS crosses; "ok-missing" otherwise
```

**Predicted unlock surface:**
- ✅ jest-worker via Next.js dev (E1's headline goal)
- ✅ webpack's parallel-terser, css-minimizer, thread-loader (because they all use jest-worker under the hood)
- ✅ Browser-rendering-style worker pools that use IPC for orchestration
- ✅ npm packages that use `child_process.fork` with rich-typed messages (vitest's worker pool, mocha's parallel runner, @swc/core's plugin runner, etc.)
- ⚠ NOT `worker_threads.Worker` directly (that's a separate API still stub-thrown)

**Effort:** 3-5 days. ~300-400 LOC across `facet-process.ts`, `supervisor-rpc.ts`, `node-shims.ts`, plus a CBOR encoder (~150 LOC) or shim atop `JSON.stringify` + Buffer/Date/Map projection (~80 LOC).

**Risk surface:**
- WebSocket-through-RPC handoff may not work — verify with prototype probe A1 first
- WebSocketPair frame ordering vs RPC ordering — prove with probe A2
- W9 hibernation: facet child WebSocket end on hibernate — re-attach on wake, same as W9 process-logs

**Verdict:** ✅✅ **Recommended.** Smallest substrate change for largest unblock; leverages existing W8/W9 plumbing.

## B.2 Coalesce all "fork siblings" into one pre-spawned facet pool

**Strategy:** Instead of spawning a real LOADER.get child per `child_process.fork(modulePath, ...)`, dispatch all forks to a **shared, pre-warmed facet pool** that runs them sequentially or with limited concurrency. Each "fork" is actually a queued task on the pool.

**The W11.5-E2 plan picks this for webpack-in-facet** (audit/sections/W11.5-E2-plan.md §0, "Option A (coalescing pool)"). Independent decision — E2 coalesces *webpack's* forks into one build-worker facet. E1 is asking whether *all* forks should be coalesced as a substrate-wide pattern.

**As fork IPC for jest-worker:** The pool worker would receive `[CHILD_MESSAGE_INITIALIZE, ...]` per task and respond with `[PARENT_MESSAGE_OK, ...]`. Same protocol as B.1, just multiplexed. The `WORKER_ID` env would be virtualized.

**Pros:**
- Lower spawn overhead for short-lived forks (the pool is warm)
- Bounded resource usage (no `fork()` flood blowing through the W8 facet pool cap)

**Cons:**
- Loses **real** parallelism. jest-worker's whole *point* is concurrent execution. A single pool worker doing them sequentially is jest-worker's fallback "no concurrency" mode — which works correctness-wise but is slow.
- ⚠ The pool worker accumulates state (loaded modules, V8 heap from previous tasks). jest-worker assumes a fresh process per worker; coalescing breaks isolation guarantees that the user's worker code may rely on (global state pollution, leaked file handles, etc.)
- ⚠ **terser-webpack-plugin specifically** spawns N workers to *parallelize* terser passes over chunks. Coalescing makes the build N× slower.

**Verdict:** ⚠️ Useful as a *fallback* (when system is under memory pressure) but NOT the right default for fork() emulation. **B.1 should be the default; B.2 is a degraded mode.**

For E1 build, ship B.1 with a tunable `NIMBUS_CP_FORK_COALESCE=1` env that flips to B.2 under pressure. Roughly mirrors how W8 already has `_CP_FACET_DIRECT` for in-supervisor execution under similar circumstances.

## B.3 Single-isolate `worker_threads` emulation in-supervisor (sync, no real concurrency)

**Strategy:** Replace `Worker.constructor` (currently stub-throws in workerd) with a same-isolate emulation that uses `MessageChannel` (which IS available!) for postMessage. The "worker" runs synchronously in the supervisor; the user code calls into `Worker` and the messages bounce through MessagePort but execute on the same event loop.

**Why it's mentioned:** `MessageChannel` is the only IPC primitive workerd ships that handles structured-clone losslessly today. If we could pretend forks are MessageChannel pairs, we'd get free fidelity. **But:**

- **No real parallelism.** Same isolate, same event loop. terser running 4 "workers" is actually 4 tasks on a queue.
- **No memory isolation.** The "worker"'s globals leak into the parent.
- **Wrong API.** `worker_threads.Worker` exposes a *separate Node-style global* (`require`, `process`, etc.) that the parent isolate doesn't pollute. Same-isolate emulation can't satisfy this without extreme contortion (proxy globals, sandboxed eval, etc.) — and we already DO have a workerd `node:vm` module that's also stub-thrown for the relevant constructors.

**As fork IPC:** No. fork() requires distinct OS-process semantics; jest-worker checks `_child.pid` and `_child.connected` and sends SIGTERM via real signals. None of that maps to same-isolate emulation cleanly.

**Verdict:** ❌ Wrong primitive for the job. MessageChannel is useful for DIFFERENT problems (in-isolate event-driven plumbing — see e.g. how Workers AI streams responses), not for fork().

⚠ **One narrow use case where B.3 makes sense:** if a future framework uses `worker_threads.Worker` *purely* as a postMessage event-bus (no real parallelism needed), B.3 unblocks it for free. But that's not Next.js's webpack pool. Park as a separate W?-?? wave.

| Dimension | E1 facet path | Container path |
|---|---|---|
| Cold start | ~30-100 ms | ~1-3 sec |

**For E1: irrelevant.** This whole section is the long-term picture. E1 ships the workerd-side substrate.

### B.4.2 Why NOT route immediately to containers and skip E1?

Tempting question. Answer:

1. **Cold-start regression.** Today Nimbus's `npm run dev` starts in ~5 sec (incl. install). Routing through a container adds 1-3 sec cold-start per fork. For a Next dev session that spawns 4 jest-worker children, that's 4-12 sec added latency. **User-visible regression.**
2. **Cost.** Containers bill per-second of running. A facet is per-request. Idle containers (e.g. between HMR cycles) are still costing if `sleepAfter` hasn't fired.
3. **Image management.** Nimbus would need to ship a Docker image with the user's node_modules pre-installed — but node_modules contents are tenant-specific. Either (a) install at container boot (slow), (b) mount Nimbus VFS via FUSE (Sandbox SDK supports this for R2; not for SQLite VFS), or (c) per-tenant image build (hard).
4. **Debugging surface.** Containers have stronger isolation; harder to debug from supervisor side. Facets share the supervisor's DO so we can introspect.
5. **Reach.** B.4 unblocks `next build` but not `next dev` HMR latency. E1 unblocks both for the IPC-handshake gate. **E1 is the higher-ROI next step.**

So: do E1 now. File W12.x for sandbox-routing as a follow-up wave when (a) we have a heavy-fork workload that justifies the cold-start tax (e.g. full `next build` compilation), or (b) workerd-side fork emulation hits a substrate ceiling we can't lift (e.g. memory-limit when 8 jest-workers all pull node_modules into RAM).

### C.5.1 The "no fork emulator inside workerd" stance

From the verified workerd source-of-truth fetches (§A.7 table), every Node API that maps to fork-style multi-process semantics is a stub-throw. This is **deliberate** per workerd's design philosophy: workerd is single-isolate per-Worker, with isolation across Workers via the runtime, not via OS processes. Adding native fork() would either:

- **(a)** Introduce real OS process semantics into workerd → breaks workerd's "isolate-only" invariant. Complex; unlikely in any short term.
- **(b)** Add an API that *looks like* fork() but routes to LOADER.get under the hood → effectively what we're building in E1 ourselves. CF could build it, but no public RFC indicates they're working on it (verified §A.7 + §B.5).

Cloudflare's response to "I want fork in Workers" is, observably, **"use Containers/Sandbox SDK"** — see [Sandbox SDK overview](https://developers.cloudflare.com/sandbox/) which markets itself precisely for "AI agents that need to execute code, interactive development environments, data analysis platforms, CI/CD systems, and any application that needs secure code execution at the edge." Same target market as Nimbus, opposite substrate choice.

### C.5.2 What CF probably will NOT do for us

Based on the workerd design philosophy and existing TODOs:

- ❌ **Native fork() in workerd** — fundamentally at odds with the single-isolate-per-Worker model.
- ❌ **Native shared-memory between isolates** — explicitly avoided per [Workers WebAssembly docs](https://developers.cloudflare.com/workers/runtime-apis/webassembly/) ("Threading is not possible in Workers").
- ⚠ **MessagePort cross-RPC** — workerd has the TODO comment ([messagechannel.h](https://github.com/cloudflare/workerd/blob/main/src/workerd/api/messagechannel.h) line ~80) but no public timeline. ⚠ speculation: 6-12 months out at minimum.
- ⚠ **`node:v8.serialize` / `deserialize`** — workerd has the TODO comment ([v8.ts](https://raw.githubusercontent.com/cloudflare/workerd/main/src/node/v8.ts) line ~243-256: *"This is one that we can implement later"*). ⚠ speculation: similar timeline. Possibly faster because it's a smaller surface.

### C.5.3 What CF probably WILL ship eventually

Based on observed velocity (the Node API surface is filling in — ctx.exports, MessageChannel, http.Server, child_process module-shape all landed in 2025-2026):

- ✅ More node:* module shapes filling in

# §D. The Next.js / jest-worker / webpack reality check

What APIs does the Next.js dev pipeline actually use? Sourced from reading the upstream code directly via webfetch.

## D.1 jest-worker exact fork() shape

**Source:** [jest-worker `ChildProcessWorker.ts` from Jest main branch](https://raw.githubusercontent.com/jestjs/jest/main/packages/jest-worker/src/workers/ChildProcessWorker.ts) (fetched 2026-05-05).

The `initialize()` method spawns:

```ts
// jest-worker ChildProcessWorker.ts (excerpt, lines ~125-145)
const options: ForkOptions = {
  cwd: process.cwd(),
  env: { ...process.env, JEST_WORKER_ID: String(this._options.workerId + 1), ...forceColor },
  execArgv: process.execArgv.filter(v => !/^--(debug|inspect)/.test(v)),
  // default to advanced serialization in order to match worker threads
  serialization: 'advanced',
  silent,
  ...this._options.forkOptions,
};
this._child = fork(this._childWorkerPath, [], options);
```

Key facts confirmed from source:
1. **`serialization: 'advanced'` is hardcoded** — jest-worker never uses JSON IPC.
2. **`silent: true` by default** — the parent doesn't pipe stdout/stderr through; it captures them via `merge-stream`.
3. **`execArgv`** is filtered. Nimbus's facet ignores execArgv entirely; that's fine.
4. **First message is** `[CHILD_MESSAGE_INITIALIZE, false, this._options.workerPath, this._options.setupArgs]` — a typed array tag + workerPath string + setupArgs (potentially Buffer-bearing).
5. **`this._child.send(request, callback)`** — uses Node's IPC channel directly, with the `serialization: 'advanced'` mode.
6. **`this._child.on('message', this._onMessage.bind(this))`** — receives `ParentMessage` arrays, also serialization-advanced.

Per [Node.js child_process serialization docs](https://nodejs.org/api/child_process.html#advanced-serialization): `serialization: 'advanced'` uses `v8.DefaultSerializer`. Equivalent value-set to `structuredClone`.

## D.2 What jest-worker actually puts in the message payload

From the same file:
```ts
this._child.send([
  CHILD_MESSAGE_INITIALIZE,
  false,
  this._options.workerPath,
  this._options.setupArgs,
]);
```

`setupArgs` is whatever the user passes. For **terser-webpack-plugin** (which webpack's optimizer uses), setupArgs is the terser config — plain object with Buffer fields possible (e.g. source maps as Buffer). For **css-minimizer-webpack-plugin**, setupArgs is the cssnano config — plain object.

For **method calls** post-init, the message shape is `[CHILD_MESSAGE_CALL, isReverse, methodName, args]`. The `args` array is whatever the user passes — Buffer-bearing in many real-world cases. Specifically, terser-webpack-plugin sends `[chunkName, sourceCodeAsString, sourceMapAsObject, ...]` where `sourceCodeAsString` is a string (fine for JSON) but `sourceMapAsObject` may include `mappings` as a Buffer-equivalent VLQ-encoded blob.

## D.3 What W8 Phase 1 misses

Per [`src/node-shims.ts:1556-1598`](../../src/node-shims.ts) (W8 fork shim) and [`audit/probes/w8/functional/cp-fork-ipc.mjs`](../../audit/probes/w8/functional/cp-fork-ipc.mjs):

```js
child.send = function(msg) {
  if (!child.connected) return false;
  if (!child.stdin) return false;
  try {
    const line = JSON.stringify(msg) + "\\n";
    child.stdin.write(line);
    return true;
  } catch (e) {
    return false;
  }
};
```

**JSON-serialize-via-`JSON.stringify`.** This:
- Buffer → `{type:'Buffer', data:[1,2,3,...]}` (verified in the W8 probe)
- Date → ISO string
- Map / Set → `{}` (silent loss)
- TypedArray → `{0:1, 1:2, length:N}` or similar
- Errors → `{}` if not specifically projected
- Circular refs → throws

When the parent (jest-worker controller) reads the response, it expects a `[type, ...]` array where `type` is a number. If the response Buffer projects to `{type:'Buffer', data:[]}`, the parent reads `response[0]` and sees a Buffer-shaped object instead of `PARENT_MESSAGE_OK = 0` — type confusion, crash.

**This is W11.5-retro §3.E1's core finding.** The W8 IPC projection is JSON-only; jest-worker assumes structured-clone equivalence; mismatch is fatal.

## D.4 Can W8 Phase 1 be uplifted with `process.send` + `process.on` shims?

**Yes, partially.** W8 already shims `process.send` (in the child runtime, when `NIMBUS_FORK_IPC=1`) and `process.on('message')` (parent side). The W8 retro §3 item 2 says: *"fork IPC v8.serialize parity. Investigate whether workerd's `node:v8` exposes `serialize`/`deserialize`. If yes, route parent.send / process.send through it for Buffer/Date/Map/Set round-trip parity."*

This research answers W8.5 item 2: **`node:v8.serialize` is a stub that throws.** So we can't use it. BUT:

- **`structuredClone()` global IS supported** ([Cloudflare changelog 2021-11-19](https://developers.cloudflare.com/workers/platform/changelog/historical-changelog/) shipped this) and is the **same algorithm** v8.serialize emits for V8-internal-format wire bytes. The **serialized form** isn't the same (structuredClone returns a deep-cloned JS value, not bytes), but for CROSS-isolate transport we never need the bytes — we need the JS value to arrive on the other side intact, which is what RPC structured-clone-arg gives us.

So the W8.5 fix path becomes:

```
W8 today:                   child.send(msg) → JSON.stringify → stdin queue
W8.5 with E1 substrate:     child.send(msg) → port.postMessage(msg)
                                            → other end receives structured-clone of msg
                                            → no JSON, no projection
```

The wire goes through either:
1. RPC method per-message: `env.SUPERVISOR.cpForkSend(childPid, msg)` where msg is RPC structured-clone'd. Round-trip: ~1ms.
2. WebSocketPair where one end is in supervisor, other is in child, and both call `port.send(structuredClone(msg))` — but ⚠ **structuredClone returns a value, not bytes**, and WebSockets need bytes/strings. So we'd need to encode the cloned value to bytes anyway. **Hence: prefer (1) for simplicity** (RPC arg is already auto-cloned; no manual byte encoding needed). Or build a CBOR encoder for B.1 binary frames.

**Cleanest E1 architecture:** RPC-call-per-message via `cpForkSend(childPid, msg)`. The RPC argument is structured-clone'd by the runtime, arrives at the supervisor as a deep-clone, supervisor pushes it onto the child's queue (already exists in [`src/facet-process.ts:cpReadStdin`](../../src/facet-process.ts) — generalize to a typed-message queue not just a string queue), child runtime delivers to `process.on('message')` listeners.

This is **simpler than B.1's WebSocketPair** because:
- We already have the RPC channel
- We already have the per-child stdin/stdout queues
- One-line change to send: `env.SUPERVISOR.cpForkSend(pid, msg)` instead of `env.SUPERVISOR.cpStdinWrite(pid, JSON.stringify(msg)+'\n')`
- The RPC arg gets structured-cloned for free

**Latency penalty:** One RPC round-trip per message. jest-worker is ~1 message per task. Each round-trip is ~1ms (in-process loopback). For a 100-task webpack run, ~200ms overhead total — acceptable.

**Throughput penalty:** RPC is rate-limited by per-request subrequest count (1000 per request on paid plan). For a single webpack build issuing ~5 messages per chunk × 1000 chunks = 5000 messages, this exceeds the RPC subrequest cap. **Need to batch messages.** Solution: don't call cpForkSend per message; let the child runtime accumulate messages in a queue, flush on next event-loop tick.

⚠ **Edge case:** synchronous-looking `child.send(msg)` followed immediately by `child.send(msg2)` — both must arrive in order. If we batch on event-loop tick, ordering is preserved (single-threaded JS). Just don't fire-and-forget.

## D.5 Can webpack be fully unblocked just by E1?

**No.** E1 unblocks the *jest-worker entry path*. But webpack's worker tasks themselves do things workerd can't:
- Loaders that use `fs.realpathSync` — workerd's `node:fs` is mostly stub; Nimbus shims via VFS but only for some paths.
- Loaders that resolve from `process.cwd()` — Nimbus has a fake cwd.
- Webpack's loader-runner uses dynamic require chains that don't pre-bundle.
- terser uses worker_threads as a fallback if jest-worker fails (W11.5-E2-plan §0 R0 step S8).

So **E1 is necessary but not sufficient for `next dev`**. E2 (webpack-in-facet) handles the rest. But E1 is the gatekeeper: until messages can carry Buffer/Date/Map without corruption, even the entry-path crashes immediately with `Cannot read property 'apply' of undefined` (W11.5-E2-plan §1.A R3 quote).

## D.6 What about `worker_threads` directly?

**Out of scope for E1.** Per [workerd `worker_threads.ts:Worker.constructor`](https://raw.githubusercontent.com/cloudflare/workerd/main/src/node/worker_threads.ts): stub-throws. Even if E1 ships and fork() works, terser's *fallback* path to `new Worker(workerPath)` will still throw. Per W11.5-E2-plan §0 R0 step S8: "terser/babel may delegate to `worker_threads.Worker` fallback" — that fallback also requires its own substrate work.

**For E1 build:** ship fork() emulation. If terser fallbacks to worker_threads (which it does only when jest-worker fails to start), we'll see a separate, distinct error — that's W11.6 / E1.5 territory.

### D.7.2 What W8 Phase 1 does today (broken)

W8 path under the hood:

| # | Step | Where | Effect |
|---|---|---|---|
| 1 | `cp.fork('/path/to/worker.js', [], opts)` from user code | supervisor `node-shims.ts:fork` | Calls `_spawn('node', [modulePath, ...])`. Returns ChildProcess emitter. |
| 2 | Inside `_spawn`: `env.SUPERVISOR.cpSpawn(req)` over RPC | supervisor | Allocates a `ChildEntry` in `FacetProcessManager.children`. Returns `{childPid: N}`. |
| 3 | LOADER.get + ctx.facets.get + facet.run | supervisor → child facet | New isolate boots. Child's preamble sees `NIMBUS_FORK_IPC=1`. |
| 4 | Parent calls `child.send([CHILD_MESSAGE_INITIALIZE, ...])` — first IPC message | supervisor | `child.send` does `JSON.stringify(msg)+'\n'`, calls `env.SUPERVISOR.cpStdinWrite(N, line)` |
| 5 | Supervisor stores the line in `children.get(N).stdinChunks` | supervisor `facet-process.ts:cpStdinWrite` | Wakes any `cpReadStdin` waiter |
| 6 | Child's runtime long-polls `cpReadStdin(N, 25_000)` | child facet | Receives `{ data: line, ended: false }` |
| 7 | Child parses `JSON.parse(line.trim())` and emits `process.emit('message', parsed)` | child facet | jest-worker's child runtime expects `[CHILD_MESSAGE_INITIALIZE = 1, ...]` |

**Where it breaks (W8 Phase 1):**

The user's `setupArgs` array can contain Buffer fields (terser config has source-map data, file-content snapshots, etc.). When `JSON.stringify` hits a Buffer:

```js
// JSON.stringify({ data: Buffer.from('hello') }) =>
// '{"data":{"type":"Buffer","data":[104,101,108,108,111]}}'
```

The child receives `[1, false, '/path/...', { data: { type: 'Buffer', data: [104,101,108,108,111] } }]`. **The `setupArgs` field is now a wrapped object instead of a Buffer.** Inside the child, jest-worker's worker runtime calls `userRequire(workerPath, setupArgs)` and the user's worker setup function does `setupArgs.data.readUInt32BE(0)` — fails with `setupArgs.data.readUInt32BE is not a function`.

Even if setupArgs is plain (no Buffer), the **response path** breaks. terser's worker.transform returns `{ code: string, map: object, ...maybe sourceMap as Buffer }`. The child sends:
```js
process.send([PARENT_MESSAGE_OK = 0, terserResult]);
```

W8 path: `JSON.stringify`. If `terserResult.map` is a Buffer-shaped property in the source map (e.g. binary VLQ for mappings field, though usually a string), it projects. The parent `_onMessage` receives `[0, { code, map: { type:'Buffer', data: [...] } }]`, expects `terserResult.map` to be JS-object — gets a wrapped Buffer object. Subsequent webpack consumer code that does `Buffer.isBuffer(map)` returns false. **Silent corruption.**

But the more catastrophic failure: jest-worker's `_onMessage(response)` does `if (!Array.isArray(response)) return; switch (response[0]) { case PARENT_MESSAGE_OK: ... }`. If our response itself wraps badly (e.g. `Buffer.from(...).toJSON()` gets called somewhere in the path), `response[0]` may not be a number. Default branch runs (silent ignore). The promise the parent awaited never resolves. **Hung worker.**

This is **exactly** the failure W11.5-E2-plan §1.A R3 quotes.

### D.7.3 What E1 (this research's recommendation) does

Same path but step 4 + 7 change:

| # | Step | Where | Effect |
|---|---|---|---|
| 4' | Parent calls `child.send([CHILD_MESSAGE_INITIALIZE, ...])` | supervisor `node-shims.ts:fork` (E1 patched) | `child.send` queues msg in __pendingIpcSends, microtask flushes → `env.SUPERVISOR.cpForkSend(N, [1, false, path, setupArgs])` |
| 5' | Supervisor stores msg in `children.get(N).ipcQueue` (NEW field, separate from stdinChunks) | supervisor `facet-process.ts:cpForkSend` | `value` is a deep-clone of the array (RPC structured-clone'd by runtime). Buffer is deep-clone'd as Buffer. Wakes any `cpForkRecv` waiter. |
| 6' | Child's runtime long-polls `cpForkRecv(N, 25_000)` | child facet | Receives `{ value: [1, false, path, setupArgs], ended: false }` — Buffer intact. |
| 7' | Child runtime delivers to `process.on('message', cb)` listeners with `value` directly — no JSON.parse | child facet | jest-worker's child receives the array intact. |

The reverse path (child → parent) mirrors:
- Step A: child runtime's `process.send(reply)` queues, flushes via `env.SUPERVISOR.cpForkSendFromChild(parentPid, childPid, reply)`.
- Step B: supervisor stores in `children.get(N).outgoingIpcQueue`.
- Step C: parent's `cpForkRecvAtParent` long-poll wakes, runtime emits `'message'` event on the ChildProcess with the value intact.

**Buffer fidelity is preserved at every step.**

### D.7.4 Failure-mode comparison table

| Scenario | W8 Phase 1 | E1 |
|---|---|---|
| Plain object IPC msg (`{n: 1}`) | ✅ works | ✅ works (faster: no JSON parse) |
| Buffer field in IPC msg | ❌ projects to `{type:'Buffer',data:[…]}` | ✅ Buffer arrives as Buffer |
| Date field | ❌ projects to ISO string | ✅ Date arrives as Date |
| Map / Set field | ❌ projects to `{}` (silent loss) | ✅ Map / Set arrives intact |
| RegExp | ❌ projects to `{}` | ✅ intact |
| Error | ❌ projects to `{}` (silent loss) | ✅ intact (with `enhanced_error_serialization` flag, default 2026-04-21) |
| TypedArray (Uint8Array) | ❌ projects to `{0:…, length:N}` (slow, lossy) | ✅ intact |
| Recursive references | ❌ JSON throws "Converting circular structure" | ✅ structured-clone handles cycles |
| Functions | ❌ JSON drops them | ❌ structured-clone also throws (same behavior — fail-fast is correct) |
| RpcStub | ❌ JSON drops | ⚠ structured-clone may handle if registered as transferable; needs probe |
| Throughput (msgs/sec) | ~5000 (RPC subrequest cap) | Same — bounded by per-request RPC quota |
| Latency per msg | ~1ms (one cpStdinWrite RPC) | ~1-25ms (long-poll round-trip) |

The only behavioral degradation is **latency per message**. Rationale: long-poll trades latency for substrate simplicity. Mitigation: if benchmarks reveal a hot path, swap to WebSocket-pushed (B.1 binary variant) without changing the visible `process.send` API.

### D.7.5 Subrequest budget math

A Worker request has a 1000-subrequest cap on paid plan ([Workers limits docs](https://developers.cloudflare.com/workers/platform/limits/)). **Both** parent's `cpForkSend` and child's `cpForkRecv` are subrequests crossing the LOADER boundary. Worst case for jest-worker:

- 4 workers × ~1 INITIALIZE message + 1000 `transform` calls per build = 4004 send + 4004 recv = **8008 subrequests**.

**This exceeds the 10,000 paid-plan cap during a `next build`.** ⚠ Direct cause for concern.

**Mitigations:**
1. **Batching at the child runtime.** Coalesce multiple sends within a microtask into one RPC call carrying an array. Reduces send-side cap usage by ~10×. Already in the §E.1 sketch (`__pendingIpcSends` flushes on microtask).
2. **Long-poll instead of short-poll on recv.** Each `cpForkRecv` returns up to 25 sec of accumulated messages, not one per call. With batching, ~100-500 messages per RPC round-trip is typical. ~10× reduction on the recv side.
3. **Per-request paid-plan limit raise.** Workers paid plan supports up to 10,000 subrequests per request, configurable via `subrequest_limit` to higher tiers. [Workers limits](https://developers.cloudflare.com/workers/platform/limits/#subrequests). For Nimbus's `next build` use case, may need to negotiate higher cap with Cloudflare.

**With both mitigations:** subrequest count for a 4-worker × 1000-task build drops to ~80 send-batches + ~80 recv-batches = 160. Well under cap. **Good.**

⚠ But for `next dev` HMR (which is the W11.5-E1 acceptance gate, not `next build`), the message rate is much lower — typically ~10-100 messages per HMR cycle. Subrequest budget is not a concern. **For `next dev`: E1 fits comfortably.**

### D.7.6 What about the `[silent: true]` jest-worker option?

jest-worker hardcodes `silent: true` (from the source we fetched at D.1):

```ts
const silent = this._options.silent ?? true;
// ... silent flag passed to fork() options
```

This means **the parent doesn't pipe stdout/stderr** — they're collected internally via `merge-stream`. Net effect: under E1, the child's `stdout` queue still fills (Nimbus's existing W8 surface), but jest-worker's parent reads via `child.stdout` pipe rather than direct callback. ProcessLogStore (W9) tees stdout to `logs <pid>`, which is what we want anyway.

**No additional work for E1** — the existing W8 stdout pipe machinery serves jest-worker's silent-mode just fine.

# §B-supplement. Decision matrix expanding §B.1

For the build wave that consumes this document, the picked option (B.1 long-poll-RPC variant) admits a few sub-decisions.

## B-supp.1 Channel separation: shared stdin queue vs separate IPC queue?

W8 today multiplexes IPC over the stdin/stdout queues. New IPC messages share the same byte stream as terminal input. **Pro:** simpler — one queue per direction. **Con:** if user code does `process.stdin.write` for a bytewise terminal write, those bytes get parsed as JSON IPC (which fails silently for non-JSON bytes — see [W8 fork shim line ~1585](../../src/node-shims.ts) "non-JSON lines are dropped silently").

**Recommendation:** **Separate queue.** Child runtime's `__supervisorIpcQueue` distinct from `__stdinChunks`. Cleaner semantics; matches Node's separate `stdio[3]` IPC channel.

## B-supp.2 RPC method naming

Three reasonable patterns:

**(a)** Add to existing `cpStdinWrite` etc. — `cpStdinWrite` carries strings, new `cpForkSend` carries structured-clone values. **Recommended.**

**(b)** Type-tag the existing `cpStdinWrite` — overload it to carry both string (treated as bytes) and array/object (treated as IPC). Confusing; bad.

**(c)** Introduce a `cpChannel(childPid, channelName, value)` generic. Future-proofs for additional IPC channels. **Over-engineered for E1.**

**Pick:** (a). Plain pair of new RPC methods.

## B-supp.3 Wait timeout: 25 sec vs longer?

W8's existing long-poll uses `Math.min(waitMs, 5000)` (5 seconds). For E1, what's right?

- **Too short** (<5s): more spurious wake-ups (resolves with `value: null`), more RPC traffic from idle children.
- **Too long** (>30s): hits Workers per-request-CPU cap on the supervisor (30s default).

**Recommendation:** 25s long-poll. Matches "below 30s default CPU cap with margin." Already set in W8's existing `cpReadStdin`. Keep symmetric.

## B-supp.4 Backpressure / queue size cap

If parent sends 10,000 messages but child hasn't drained any, the supervisor's `children.get(N).ipcQueue` grows unbounded. Memory bomb.

**Recommendation:** Cap at 1000 messages or 16 MiB (whichever first). Beyond cap, `cpForkSend` returns `{ ok: false }`. Parent's `child.send` returns false (Node-API parity — `child.send` returns boolean). User code can react.

```ts
// audit-only sketch
async cpForkSend(childPid, value) {
  const child = this.children.get(childPid);
  if (!child) return { ok: false };
+ if (child.ipcQueue.length >= 1000) return { ok: false };
+ // size estimate: rough; for production maybe encode-then-measure
+ if (child.ipcTotalBytes + estimateSize(value) > 16 * 1024 * 1024) return { ok: false };
  // ... existing enqueue
}
```

## B-supp.5 Disconnect semantics

Node's `child.disconnect()` ends IPC but keeps the process running. Our E1: `child.disconnect()` should set `child.connected = false`, drain pending sends, **stop** the recv loop, **NOT** kill the child. Mirrors Node.

`process.disconnect()` from inside the child: same pattern — local `process.connected = false`, stop the recv loop, no exit triggered.

## B-supp.6 What if RPC structured-clone has gaps we didn't anticipate?

**Detection:** probe A1+A2 (write before build) verify Buffer/Date/Map/Set/RegExp/TypedArray/recursion all round-trip. Run against local wrangler-dev FIRST, then prod.

**If a gap is found:** revert to JSON projection for that specific type, with a clear warning logged. `structuredClone` is the default; type-class allowlist enforces fidelity.

**Most likely gaps to test:**
- `BigInt` — structured-clone supports; verify
- `BigInt64Array` / `BigUint64Array` — structured-clone supports; verify
- Custom class instances with prototype — structured-clone produces plain object on receive; expected. jest-worker doesn't pass class instances over IPC.
- `URL` — structured-clone supports; verify
- `URLSearchParams` — structured-clone supports; verify

Each of these gets a probe in A1-A5 corpus.

## B-supp.7 Sequencing — does cpForkRecv need to be in-order?

Yes. jest-worker's protocol assumes in-order delivery (`PARENT_MESSAGE_OK` is paired with the most recent send). FIFO queue gives that for free as long as sends from one parent are serialized through one RPC invocation at a time.

**Edge case:** if the parent fires two `cpForkSend` calls without awaiting the first, the order on the supervisor side depends on RPC arrival order. **Workerd RPC is in-order per-stub** (verified by [RPC docs](https://developers.cloudflare.com/workers/runtime-apis/rpc/) — calls on a single stub are serialized). So `await env.SUPERVISOR.cpForkSend(pid, m1); await env.SUPERVISOR.cpForkSend(pid, m2);` is in-order, AND `env.SUPERVISOR.cpForkSend(pid, m1); env.SUPERVISOR.cpForkSend(pid, m2);` (no await) is also in-order because the RPC system serializes per-stub.

The microtask-batched flush in §E.1 sends them as separate calls on the same stub — in-order. ✓

## B-supp.8 Dual-mode: legacy JSON vs new structured-clone

For the W8 regression test to keep passing, the new code must support BOTH:

```ts
// audit-only sketch
const childEnv = {
  ...,
  NIMBUS_FORK_IPC: opts.serialization === 'json' ? '1' : 'v8',  // new opt; default 'v8'
};
```

Child runtime selects the wire path based on env var:
- `NIMBUS_FORK_IPC=1` (W8 legacy): JSON-newline over stdin queue
- `NIMBUS_FORK_IPC=v8` (E1 default): structured-clone over cpForkSend/Recv

Allows W8 regression to keep asserting the JSON projection by passing `serialization: 'json'`. Allows new code (jest-worker, etc.) to use structured-clone implicitly.

⚠ The Node `fork(modulePath, [], { serialization: 'json' })` option already exists — Node lets users opt out of advanced for legacy compat. Honor that flag in our shim. Default `'advanced'` per Node docs since Node 13+.

**Why this works:**
- RPC arguments are structured-clone'd by the runtime — Buffer, Date, Map, Set, RegExp, ArrayBuffer, TypedArray, recursive references all round-trip.
- jest-worker's `[CHILD_MESSAGE_INITIALIZE, false, workerPath, setupArgs]` arrives as exactly that array on the child side. workerPath is a string, setupArgs is whatever object the user passed — both round-trip via structured-clone.
- Errors round-trip with the `enhanced_error_serialization` flag — already default at compat date `2026-04-21` per [Workers compat flags](https://developers.cloudflare.com/workers/configuration/compatibility-flags/).
- Total substrate change is bounded and testable.

**Why long-poll vs push:**
- Long-poll on `cpForkRecv` mirrors Nimbus's existing W8 pattern for stdin (`cpReadStdin`).
- Alternative: push via WebSocket-as-IPC (B.1 original variant). Adds a substrate (WebSocketPair-via-RPC) we haven't verified. Long-poll uses only RPC, which we know works.
- ⚠ Long-poll trades latency for substrate simplicity. Per-message latency is ~1-25ms (whatever's left of the 25s long-poll). For jest-worker that's usually fine — initialization is one batch, then per-task is one round-trip. **But:** if a user's IPC pattern is "1000 small messages per second", long-poll is the wrong shape; need WebSocket. Defer to E1.5 if needed.

## G.4 Rollback plan + risk register

### G.4.1 If E1 ships and breaks W8 regression

- Revert mechanism: `NIMBUS_FORK_IPC=v8` → `NIMBUS_FORK_IPC=1` env var flips back to JSON projection. **No code revert needed**, just env-var control.

- Branch protection: keep W8 probes in CI. Any new breakage on regression = hard block.

### G.4.2 If RPC structured-clone has a fidelity gap we missed

- Detection: probes A1-A4 run before E1 build dispatch. If a gap is found, **don't ship that type** — fall back to a JSON projection for that specific type, log a warning.
- Most likely gap risks: BigInt edge cases, custom prototype loss (already known structured-clone behavior), TypedArray buffer-detach semantics.
- Mitigation: typed allowlist enforcement: a runtime check rejects unsupported types with a clear error, not a silent corruption.

### G.4.3 If subrequest budget proves fatal at scale

- Probe early on prod with a real Next dev session. Measure RPC count.
- If hot: implement microtask batching aggressively (multiple sends per RPC call). 10× margin available.
- Worst case: file with Cloudflare for higher subrequest cap (Workers Enterprise paid plan can negotiate).

### G.4.4 If WebSocketPair-via-RPC fails (alternate path B.1 binary)

- Not a risk for picked option (long-poll RPC); only relevant if we pivot to WS path.
- Mitigation: probe A0 (WebSocket-through-RPC handoff) gates the pivot. If A0 fails, stay on long-poll.

### G.4.5 Hibernation interaction (W9)

- Long-poll RPC: child runtime is "blocked" on `cpForkRecv`. If supervisor hibernates, the recv-loop's RPC promise is in flight. On wake, it should auto-resume (RPC stub semantics).
- ⚠ Verify with probe: hibernate + wake while a `cpForkRecv` is pending. Result must be the message arriving normally OR a clean `ended:false, value:null` allowing the loop to retry.
- Worst case: child's recv loop loses sync on wake. Mitigation: detect via heartbeat (parent-to-child periodic NOOP message); if missed, child re-establishes.

### G.4.6 Memory pressure from queued IPC messages

- `children.get(N).ipcQueue` and `outgoingIpcQueue` accumulate JS values. Each Buffer in the queue holds a copy.
- If parent sends 1000 messages each carrying a 16 MiB Buffer, supervisor has 16 GiB pinned in memory — way over DO 128 MiB cap.
- Mitigation: per B-supp.4 cap (1000 messages OR 16 MiB total per direction). Beyond cap, send returns false (Node-API parity).

### G.4.7 Incompatible packages we know about
