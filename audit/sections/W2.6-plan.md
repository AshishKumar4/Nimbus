## 1. Background

W2.5b (HEAD 6a128f5) decoupled directory **shape** from content cap by
introducing `__vfsManifest` (uncapped path→child-names map) alongside
the capped `__vfsBundle` (path→content). `fs.readdirSync` from the
facet now sees ~all installed packages. But content for transitive
deps still hits the cap, so `require()` of capped-out files fails:

```
$ npm install fastify
added 46 packages (1928 files)
$ node -e "require('fastify')"
Error: Cannot find module 'fastq' (from .../avvio)
```

W2.5b retro proposed three architectural moves; W2.6 picks the two with
combined +N-packages return-on-effort and bounded blast radius.

## 2. Area A — Workerd module budget

### 2.1 Documented limits (Cloudflare docs, fetched 2026-05-03)

Sourced via the Cloudflare Docs MCP server during plan-mode discovery
(no separate probe artifact — these are vendor-published numbers):

| Limit | Workers Free | Workers Paid |
|---|---|---|
| Worker size after gzip | **3 MB** | **10 MB** |
| Worker size before compression | **64 MB** | **64 MB** |
| Memory per isolate | 128 MB | 128 MB |
| CPU time per request (default) | 10 ms | 5 min |
| Worker startup time | **1 second** | **1 second** |
| RPC payload (serialized) | **32 MiB** | **32 MiB** |
| RPC stream payload | unbounded (ReadableStream) | unbounded |

Sources:
- https://developers.cloudflare.com/workers/platform/limits/
- https://developers.cloudflare.com/workers/runtime-apis/rpc/ ("The maximum serialized RPC limit is 32 MiB")
- https://developers.cloudflare.com/dynamic-workers/getting-started/ (LOADER.get/load API)
- https://developers.cloudflare.com/dynamic-workers/api-reference/ (`modules: Record<string, string | Module>`)

### 2.2 Empirical evidence from W2.5b deploy thrash

Plan-mode design choice (per operator answer): binary-search the limit
via existing prod artifacts rather than burning fresh deploy slots. The
W2.5b round of deploys (commit 6a128f5 chain) already swept the cap
across multiple values, and the artifacts under
`audit/probes/regression/install-pipeline-coverage.txt` and
`audit/probes/w25-diag-fastify*.txt` record each transition.

| `VFS_BUNDLE_MAX_BYTES` | fastify install (1928 files / 1.3 MiB tarball) | Outcome |
|---|---|---|
| 32 MiB | facet boot fails immediately | `wsClosed=true` on first run |
| 12 MiB | facet boot fails immediately | `wsClosed=true` |
| 8 MiB | facet boot intermittent | `wsClosed=true` ~50% |
| **6 MiB** | **boots OK, all 41 packages visible** | **PASS** in regression |
| 5 MiB | boots OK, fastify 3/4 PASS but content-truncates | drizzle-orm regresses |
| 4 MiB | boots OK, conservative | shipped value (current prod) |

**Verbatim error captured at 12 MiB cap** (extracted from the W2.5b
fastify diag run, file `audit/probes/w25-diag-fastify2.txt`):

```
[facet started: pid=2 cmd="node /tmp/p_1_molmom35.js"]

---- step done in 1502ms (wsClosed=true) ----
```

The session WebSocket closes immediately when the facet attempts to
boot a worker module exceeding the budget. No exception surfaces in
`wrangler tail` (the worker never reaches the request handler — the
LOADER itself rejects the module).

**Verbatim error captured at 32 MiB cap**, after a second deploy to test
a higher ceiling:

```
[31m✘ [41;31m[[41;97mERROR[41;31m][0m
A request to the Cloudflare API
(/accounts/.../workers/scripts/nimbus/versions) failed.

  Uncaught SyntaxError: Unexpected token ':'
    at null.<anonymous> (index.ts:52:31)
   [code: 10021]
```

This was actually a `--no-bundle` deploy artifact (raw TS uploaded), not
the LOADER limit. After the bundled deploy succeeded with 32 MiB content
cap, fastify fixtures hit `wsClosed=true` on facet boot — the LOADER
rejection surface (no exception text reaches `wrangler tail`).

The 6→8 MiB transition is much sharper than the documented 10 MB-gzip
parent-Worker ceiling. The plausible mechanism (corroborated by
sub-agent challenge): the bundle goes through `JSON.stringify` →
embedded as a string literal in the worker module text. JSON-escaping
inflates JS source ~3-5× (newlines as `\n`, quotes as `\"`,
control chars as `\u`, etc). A 6 MiB raw bundle becomes ~20-25 MiB of
module text source; gzip compresses that to ~3-5 MiB. **The actual
ceiling is the per-module text-size budget after JSON encoding, not the
raw byte sum.** No public Cloudflare doc states the LOADER per-module
text-size number explicitly.

### 2.3 Verdict on Area A

**The current cap dimension is wrong.** `VFS_BUNDLE_MAX_BYTES` measures
raw content; the real budget is `Buffer.byteLength(JSON.stringify(bundle))`.
A correct cap variable would gate on the latter — see W2.6a §4 for the
fix.

For W2.6 architectural choice: any approach that pumps a single
JSON-stringified blob into the LOADER `modules` Record is bounded by
~5 MiB raw / ~20 MiB JSON-encoded. Anything larger requires either
(a) splitting across multiple `modules[path]` entries (workerd compiles
each separately), or (b) lazy-loading content via a stream (works for
data, not for executable code due to request-time eval block — see §3
B3 verdict).

## 3. Area B — Five options surveyed

### B1 — Static-resolve dependency prefetch ⚠️ TRADEOFF (recommended for W2.6a)

**Concept:** Walk the user's entry point + `require()` chain server-side,
inline only the reachable transitive set. Smaller bundles, full coverage
of what user code actually needs.

**Existing scaffolding:** `src/require-resolver.ts:109` already implements
`prefetchForRequire(vfs, entryCode, cwd, entryFile?) → Record<string,
string>`. Currently quarantined (ARC-A-P1 banner per `nimbus-session.ts:1015-1026`)
but functional. Limits: `MAX_FILES = 2000`, `MAX_BYTES = 8 MiB` — close
to the W2.5b numbers.

**Coverage analysis (sub-agent challenge §Q1):**
- Regex `/require\s*\(\s*['"]([^'"]+)['"]\s*\)/g` catches >95% of CJS
  patterns in npm.
- Misses: `require.resolve(...)`, template-literal-no-interp
  ``require(`x`)``, dynamic `require(variable)`, ESM `import`.
- For our 33-pkg smoke set, regex misses are NOT the dominant failure
  mode — the failures are content-cap evictions (regex would catch the
  needed paths; the content just isn't bundled).

**Predicted delta (sub-agent §Q6) per failing package:**

| Pkg today | Failure | B1 fixes? |
|---|---|---|
| fastify | `Cannot find module 'fastq'` | ✅ |
| redis | `Cannot find './lib/RESP/decoder'` | ✅ |
| drizzle-orm | `Cannot find '../utils/array.cjs'` | ✅ (likely) |
| express | `Object prototype may only be Object or null` | ❌ exports-conditions bug |
| ts-jest | `Cannot find 'typescript'` | ❌ typescript itself > any cap |
| framer-motion | `Cannot find 'framer-motion'` | ❌ ESM-only exports |
| axios | `Cannot find 'http2'` | ❌ builtin (W3) |
| vitest | `Cannot be imported in CJS` | ❌ ESM-vs-CJS (W3+) |
| jest, zod, pg, better-sqlite3 (4 ✅ today) | unchanged | — |

**Realistic post-W2.6a count: ~7/33** (4 baseline + 3 from B1).

**Pros:**
- Fits within current architecture; no LOADER contract change.
- `require-resolver.ts` already exists.
- Reachable set for fastify ≈ 1.5-2 MiB raw / ~8 MiB JSON-encoded.
  Comfortably under empirical ceiling.

**Cons:**
- Does NOT fix exports-conditions resolver bug (express).
- Does NOT fix oversize-package case (ts-jest+typescript ~9 MiB single file).
- Does NOT fix ESM-only resolution (framer-motion).
- Sub-agent §Q7: regex prefetch RISKS REGRESSING currently-✅ packages
  (jest's `import-local` plugin loader, better-sqlite3's `bindings`
  computed-path-load) UNLESS paired with greedy oversampling.

**Verdict:** ⚠️ TRADEOFF — recommended as W2.6a but with mandatory
mitigations (greedy oversampling, JSON-encoded cap dimension, resolver
unification). NOT a full solution to "≥10/33 ✅".

### B2 — Per-package modules-map ❌ DEAD-END as written; ✅ RECOMMENDED in revised form (B2′)

**Original framing in prompt:** every node_modules pkg as a separate
LOADER module entry, hoping the budget is per-module not aggregate.

**Refuted by docs:** Cloudflare's Worker size limit is per-script,
aggregate across all `modules[*]` entries. Splitting by package doesn't
extend the budget.

**Revised form (B2′ — multi-module SHIPPING by file):** Instead of
embedding every VFS file's content as a literal string inside one
`runner.js` module, ship each VFS file as its OWN entry in the LOADER
`modules` Record:

```js
LOADER.get(codeId, async () => ({
  mainModule: 'runner.js',
  modules: {
    'runner.js': SHIM_AND_BOOT_CODE,                // ~50 KiB
    'vfs/home/user/app/node_modules/fastify/lib/route.js': '...',
    'vfs/home/user/app/node_modules/fastq/queue.js': '...',
    // ... thousands of entries, each its own module
  },
}));
```

Each VFS file is its own workerd module. Workerd compiles each at
module-load — same `new Function()` allowance applies, but per-module.
The text-size cap that fired W2.5b was on a single JSON-stringified blob;
splitting that blob across N modules shifts the constraint to: total
script size (10 MB gzipped Paid) and per-module size (likely the same
~5 MB ceiling, but each file is ≤ 1 MiB so this is non-binding for
typical npm packages).

**Validation needed (verified empirically in W2.6b probe step 4):**
- Confirm workerd treats `modules` entries as independent compile units
  (vs concatenating). The `wasmModules` pattern (`pre-bundle-preamble.ts:103-109`)
  works this way for binary modules; verify text modules behave likewise.
- Confirm dynamic `import('vfs/path/to/file.js')` works at module-eval
  time within the dynamic worker (separate from the request-time eval
  block — module-load `import()` is allowed per workerd docs).
- Confirm top-level await inside a dynamic worker module's body
  (verified for ESM regular Workers; needs in-LOADER probe).

**Pros:**
- Sidesteps the per-blob cap entirely. ts-jest / typescript become
  feasible (typescript.js ships as one ~9 MiB module — still tight
  against the per-module limit, but no longer concatenated with 200
  others).
- Each module compiled lazily on first import; cold-start cost spread
  via parallelism.
- Workerd's content-hash dedup means repeated installs share warm
  modules across LOADER cache entries.

**Cons:**
- BIG architectural change: requires loading content via `import()` not
  `__vfsBundle[path]` lookup. Existing `node-shims.ts` `__readFileOr`,
  `__loadModule`, `_bundleLookup` need rewiring.
- 10 MB gzipped script TOTAL still applies (per-module is just a
  finer-grained slice). For very large installs (full webpack tooling),
  still tight.
- Build-mode complexity: `generateFacetCode()` must enumerate a
  `modules` Record indexed by path; the `mainModule` runner needs a
  per-import dispatch table.

**Verdict:** ✅ RECOMMENDED as W2.6b. Big lift but the right strategic
answer; B1 alone hits a cap that B2′ doesn't.

### B3 — Two-phase boot (empty facet + RPC pull) ❌ DEAD-END

**Concept:** Facet boots with TINY worker module (shims only); `run()`
first does `await env.SUPERVISOR.getReachableBundle()` to populate
`__vfsBundle`, THEN compiles + runs user code synchronously.

**The blocker (verified):**
- `audit/probes/dynamic/eval-and-Function.out.txt`: `new Function()`
  throws "Code generation from strings disallowed for this context" at
  request time inside facets (workerd's
  `disallow_eval_during_request_handler`).
- `node-shims.ts:1122` comment: "Use pre-compiled function from startup
  (new Function allowed at module eval time)".
- `audit/sections/07-workerd-hard-limits.md:39-46` documents this
  workerd hard rule.

**Verbatim probe evidence** (from `audit/probes/dynamic/eval-and-Function.out.txt`):

```
[facet started: pid=2 cmd="node /tmp/dyn_mokemila_eval-and-Function.js"]
new Function fail: Code generation from strings disallowed for this context
eval fail: Code generation from strings disallowed for this context

────────────────────────────────────────────────────────────
Process 2 (node /tmp/dyn_mokemila_eval-and-Function.js) exited with code 0
────────────────────────────────────────────────────────────
```

**Why this kills B3 as conceived:**
1. Module-eval phase: `env` not in scope. Cannot await
   `env.SUPERVISOR.getReachableBundle()` from module top-level (env is
   per-instance and only available at constructor / request-handler time).
2. Request-handler phase: `env` available, but `new Function(code)` is
   forbidden. Cannot compile fetched module text.
3. Top-level await (during module eval) doesn't have env access.

**Possible workaround:** Generate the module text DOUBLE-LOOKED: the
supervisor's first LOADER.get call runs static-prefetch synchronously,
inlines the result, returns the module text. This is identical to B1.
Two-phase boot adds round trips; B1 already does the prefetch
supervisor-side at codegen time. **B3 reduces to B1 once the eval
block is honored.**

**Verdict:** ❌ DEAD-END. The "two-phase boot" framing in the W2.5b
retro was a red herring; the architecture cannot work as imagined.

### B4 — Code-transform sync→async (rewrite `require` → `await import`) ❌ DEAD-END

**Blast radius:**
- All user code: parse + AST-rewrite every CJS module.
- All node_modules content: same. ~1928 files for fastify, ~5300+
  for ts-jest+typescript.
- Circular requires: CJS allows partial-export observation during cycle
  resolution; async-import does not (resolves at end, cycles deadlock).
  Many packages depend on this (e.g. `pg-pool`, several jest internals).
- Sync entry points: jest's `Runtime` class uses `requireModuleOrMock`
  which is sync by design. Rewriting changes Jest's plugin API.
- Performance: every require is a microtask boundary. Deep CJS trees
  (hundreds of files) become hundreds of microtasks. User-perceived
  startup latency.
- Tooling fingerprint: stack traces, `module.cache`, `require.resolve`,
  `__filename`/`__dirname` — all break or behave subtly differently.

**Verdict:** ❌ DEAD-END for W2.6 timeframe. Possible long-term as part
of an ESM-first rebuild; out of scope here.

### B5 — Per-execution-mode strategy ✅ RECOMMENDED in hybrid (§4)

**Survey of execution modes** (full table from plan-discovery sub-agent
exploring `LOADER.get` / `LOADER.load` / `ctx.facets.get` callsites):

| Mode | Today | W2.6 strategy |
|---|---|---|
| `node script.js` / `node -e` | bake-everything-in-bundle (`facet-manager.ts:854`) | **B1+B2′** |
| `vite dev` (long-running) | per-request RPC, no bundle (`facet-manager.ts:1031`) | unchanged |
| `vite build` invoked via `npm run build` | bake-everything | **B1+B2′** (same path as node) |
| `wrangler dev` | esbuild-pre-bundled user Worker (`nimbus-wrangler.ts:430,453`) | unchanged |
| `npm install` (resolver/batch facets) | per-call slice via RPC arg (`npm-installer.ts:537,623,736`) | unchanged |
| `git clone/fetch/pull/push` | buffered fs adapter via RPC (`git-network-facet.ts:94`) | unchanged |
| pre-bundle (esbuild slice) | per-spec slice via RPC arg (`npm-installer.ts:1316`, `vite-dev-server.ts:1309`) | unchanged |
| cirrus-real (alt vite) | globalThis seed snapshot (`cirrus-real.ts:607`) | unchanged |

**The bundle problem is exclusive to `FacetManager.exec()`
(`facet-manager.ts:742`). Every other dispatch site already uses an
async-friendly pattern.** This dramatically narrows the W2.6 surface.

**Verdict:** ✅ RECOMMENDED. W2.6 changes only `FacetManager.exec` and
its codegen siblings (`generateFacetCode` / `generateEntrypointCode`).

## 4. Recommended hybrid architecture

### 4.1 Stage W2.6a — bundled prefetch (B1+)

**Replace `buildVfsBundle` with `buildPrefetchBundle`:**

```ts
// src/facet-manager.ts (replaces lines 487-666)

import { prefetchForRequire } from './require-resolver.js';

function buildPrefetchBundle(
  vfs: SqliteVFS,
  cwd: string,
  entryFile: string | undefined,
  entryCode: string,
): FacetVfsState {
  // 1. Static reachable-set walk from entry — only the JS files that
  //    the user's `require()` chain hits, plus their package.json.
  const reachable = prefetchForRequire(vfs, entryCode, cwd, entryFile);

  // 2. Greedy oversampling for dynamic-require survival (jest plugins,
  //    bindings, framework hot-paths). For every installed package,
  //    add package.json + main entry + index.js if not already present.
  const greedy = greedyAddMainEntries(vfs, cwd, reachable);
  Object.assign(reachable, greedy);

  // 3. Manifest pass — UNCHANGED from W2.5b. Uncapped path→child-names
  //    so fs.readdirSync still sees full directory shape.
  const manifest = buildManifest(vfs, cwd);

  // 4. Cap on JSON-ENCODED size, not raw bytes (W2.6a fix to
  //    Concern #2 from sub-agent review).
  return capByEncodedSize(reachable, manifest, BUNDLE_MAX_ENCODED_BYTES);
}
```

### 4.2 Stage W2.6b — multi-module shipping (B2′)

**Replace single-blob bundle with per-file modules Record:**

```ts
// src/facet-manager.ts (after W2.6a)

function generateFacetCode(userCode: string, vfsState: FacetVfsState): {
  modules: Record<string, string>;
  mainModule: string;
} {
  const modules: Record<string, string> = {};

  // Each VFS file → its own Worker module entry
  for (const [path, content] of Object.entries(vfsState.bundle)) {
    if (path.endsWith('.js') || path.endsWith('.cjs') || path.endsWith('.mjs')) {
      modules[`vfs/${path}`] = wrapAsCjsModule(content, path);
    } else if (path.endsWith('.json')) {
      modules[`vfs/${path}`] = `export default ${content};`;
    }
    // Other extensions are read as TEXT through __readFileOr — embedded
    // in the runner.js __vfsTextFiles record (small, capped).
  }

  modules['runner.js'] = generateRunnerCode(userCode, vfsState.manifest);
  return { modules, mainModule: 'runner.js' };
}
```

`runner.js` then loads files via dynamic `import()` at module-load
(allowed scope) — top-level await populates a `__compiledModules`
registry before `run()` is called; sync `require()` reads from the
registry.

## 8. Open architectural decisions YOU need to answer before build mode

These need explicit operator answers before W2.6a kickoff:

### D1. Stage W2.6b is a 3-day lift. Approve scope?
**Default:** ship W2.6a (~1 day, ~7/33), then evaluate ROI before W2.6b.
**Alternative:** skip W2.6a, go straight to W2.6b — but W2.6b probes
might invalidate the approach (top-level await, modules count). W2.6a
gives a +3 sure thing while we de-risk W2.6b.

### D2. Express/framer-motion exports-conditions resolver bug — in-scope or W3?
W2.6 doesn't fix express. The bug is in `__resolvePkgSubpath`'s exports
conditions selection (chooses `polyfill.js` over `index.js` for
setprototypeof). Sub-agent §Q1 identifies it. Could pair-fix with
W2.6a's resolver-unification PR (cheap, +1 package). Default: pair-fix.

### D3. ts-jest / typescript: ship as W2.6c or punt to W3?
Even W2.6b's per-module limit might not fit `typescript/lib/typescript.js`
(~9 MiB single file). Solutions:
- (a) split typescript.js into multiple chunks via esbuild's
  `--splitting` (possible at install time via pre-bundle pipeline).
- (b) ship typescript via the existing `pre-bundle-facet.ts` slice
  pattern — but ts-jest doesn't go through vite.
- (c) accept ts-jest stays ⚠️ until W3 architectural revisit.
Default: (c) — note in W2.6b changelog that typescript-class oversize
packages need separate work.

### D4. ESM-only packages (framer-motion, vitest)
The loader is CJS-only. To turn framer-motion ✅, we'd need ESM-aware
loader changes — orthogonal to bundle architecture. W3 territory?
Default: yes, W3.

### D5. Cleanup of `/tmp/w26-probe-A.mjs`
The stray probe file from §0. Build-mode round can `rm` it as the first
gesture. Or leave it; harmless. Default: rm in W2.6a kickoff.

### D6. Resolver unification scope
Step 2 of W2.6a proposes extracting `resolvePackageEntry` from
`require-resolver.ts` into the same shared module that `node-shims.ts`
uses. This touches one runtime path (node-shims runtime require) and
two prefetch paths. Scope alone? Or fold into W2.6a?
Default: fold into W2.6a (resolver-unification is the foundational
correctness fix; without it prefetch and runtime can disagree).

### D7. Build-mode time-box for W2.6a
Sub-agent estimate: ~1 day. Mine: ~1 day if greedy oversampling
calibration takes one iteration; ~2 days if it takes two. Confirm time
budget before kickoff.
