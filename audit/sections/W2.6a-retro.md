## 3. Actual ✅ delta vs predicted

**Predicted:** +3 packages (`fastify`, `redis`, `drizzle-orm`) flip ⚠️→✅.
**Measured:** +1 package (`drizzle-orm` only).

W2.6a's bundling fix DID work — proven by failure-mode shifts on the
predicted-flip packages. But each one hit a NEW wall one layer deeper:

| Pkg | W2.5b error | W2.6a error | Wall class |
|---|---|---|---|
| fastify | `Cannot find module 'fastq'` | `Cannot find module 'node:diagnostics_channel'` | **W3 builtin shim** — fastify ≥4.x requires `node:diagnostics_channel`, which is not in `node-shims.ts` builtins table. Fastify's require chain now reaches its own first-line require statement; we crashed BEFORE that previously. |
| redis | `Cannot find './lib/RESP/decoder'` | `Cannot read module: @redis/client/dist/lib/client` | **content cap** — manifest now sees the file (W2.5b artifact), so `__fileExists` returns true. But the content was evicted by the JSON-encoded-size cap. `__readFileOr` returns null → `Cannot read module:` (vs the old `Cannot find module:`). The missing file is real, just not in the bundle. |
| drizzle-orm | `Cannot find '../utils/array.cjs'` | (none — flips ✅) | **B1+ thesis worked** — prefetch caught all transitive requires that the body-pass had been racing on. |
| puppeteer-core | `Cannot find './locators/locators.js'` | `Cannot find module 'node:fs/promises'` | **W3 builtin shim** |
| express | `Object prototype may only be Object or null` | (same) | **unchanged** — sub-agent speculation that this was an exports-conditions bug was wrong. The real cause is something interacting with the runtime require path that B1+ didn't touch (probably circular-require with reassigned `module.exports`). Out of W2.6a scope. |
| framer-motion | `Cannot find module 'framer-motion'` | `Cannot find module 'react/jsx-runtime'` | **smoke-test gap** — framer-motion now resolves; the smoke probe doesn't `npm install react` as a peer. Test-harness fix, not a Nimbus bug. |
| ts-jest | `Cannot find module 'typescript'` | (same) | **D3 deferred** — typescript package itself drops out of install pipeline (separate install-side issue). |

Net: **the +3 prediction was for "all three flip ✅"; reality is "all
three advanced one layer, only drizzle-orm cleared".** B1+ is doing
the right thing structurally — each package's failure surface moved
deeper into the require chain — but each next-layer wall is a
different unrelated bug (missing builtin shims, content-cap eviction,
test-harness peer-deps).

  synthesizes a fixture package directly into the VFS and asserts the
  fallback path. Sub-agent's first-pass review caught that the probe
  was hitting setprototypeof's HAPPY path (no exports field, so the
  fallback never fires); rewritten to use a synthetic fixture that
  guarantees the fallback path executes.
- **`SqliteVFS.readdir` lexicographic sort** added (sub-agent concern

  which is non-deterministic under concurrent install). Cheap O(n log
  n) on already-O(n) work; removes a class of "works on Tuesday" flakes.

Nothing scope-creeped beyond the plan; all extras were sub-agent-
review-driven hardening.

## 5. W2.6b ROI verdict — **SKIP for now, monitor**

W2.6b proposed multi-module shipping (one workerd module per VFS file)
to sidestep the per-module text-size limit. The case for it was that
W2.6a wouldn't fit ts-jest-class oversize installs.

**With W2.6a measured at +1 package**, W2.6b's expected delta needs
re-evaluation:

| What W2.6b is supposed to fix | Reality post-W2.6a |
|---|---|
| ts-jest's typescript.js (~9 MiB single file) > per-blob cap | typescript drops out of install pipeline entirely (D3, separate bug). W2.6b alone wouldn't fix this; we need the install fix first. |
| Big tree installs that exceed `BUNDLE_MAX_ENCODED_BYTES` | The W2.6a JSON-encoded cap is 22 MiB. fastify install fits with margin. The `redis` content-cap case shows a borderline situation but it's not the dominant failure mode in our 33-pkg set. |
| Content-cap evictions of files in the require graph | redis's `dist/lib/client` eviction is the canonical case. ONE package out of 33 is bottlenecked here. |

The dominant remaining walls are NOT bundle-size — they're:
1. **Missing builtin shims** (`node:diagnostics_channel`, `node:fs/promises`,
   `http2`, `repl`) — W3 builtin-shim work, +3-5 packages estimated.
2. **Smoke-harness peer-dep gaps** (framer-motion, radix-react-dialog
   need `react` installed alongside) — test-only fix, +2 packages.
3. **ESM-only packages** (vitest) — D4 deferred to W3.
4. **Pre-existing express object-prototype bug** — needs separate
   investigation; unrelated to bundling.

W2.6b's effort estimate was 3-5 days; expected delta now looks like
+1-2 packages (mostly redis content-cap case + maybe one more). **ROI
is poor.** Better next move: **W3** (builtin shims) which has a
clearer +3-5 trajectory at lower complexity. If W2.6b becomes
necessary later (e.g. typescript install starts working and we hit the
9 MiB single-file wall), it can be revived with the prototype probes
already designed in `audit/sections/W2.6-plan.md` §5.2.

**Recommendation: skip W2.6b. Dispatch W3 next.** Reassess W2.6b after
W3 lands and we have a fresh 33-pkg measurement.
