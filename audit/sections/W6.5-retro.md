## 3. Telemetry hook design — actual shipped vs planned

### Shipped

| Component | Where | What |
|---|---|---|
| `RegistryEvent` type | `src/wasm-swap-registry.ts` | Discriminated union: `swap` / `reject` / `transitive-skip`; ctx is `'top' \| 'transitive'` for swap+reject. |
| `RegistryEventSink` type | same | `(e: RegistryEvent) => void`. |
| `setRegistryEventSink(sink \| null)` | same | Module-level singleton. |
| `getRegistryEventSink()` | same | Inspection. |
| `emitRegistryEvent(e)` | same | try/catch around `_sink?.(e)`; throws → `_sinkThrowCount++`. |
| `getSinkThrowCount()` | same | For monitoring + probe assertions. |
| Supervisor top-level emit | `npm-installer.ts:applyW6Registry` | one `swap` (ctx top) per swap; one `reject` (ctx top) per offending package BEFORE the throw. |
| Supervisor BFS emit | `npm-resolver.ts` resolver loop | one `swap` (ctx transitive) on swap rewrite; one `transitive-skip` on warn-policy hit; one `reject` (ctx transitive) before fail-policy throw. |
| Facet preamble | `parallel/npm-resolve-preamble.ts` | new `__pendingEvents: any[]` + `__EMIT_EVENT(e)` push helper + `__DRAIN_EVENTS()` returns + clears. |
| Facet body emit | `npm-resolve-facet.ts:resolveOne` | `__EMIT_EVENT` at all three decision sites. |
| Facet result carries events | `ResolveFacetResult.registryEvents: FacetRegistryEvent[]` | New required field. |
| Supervisor drain | `npm-installer.ts:580` (after facet returns) | reads `(result as any).registryEvents` soft-failingly, forwards each via `emitRegistryEvent` (sink errors swallowed inside `emitRegistryEvent`). |
| Default sink | `src/index.ts` (module top) | `console.log(\`[w6.5/registry] ${JSON.stringify(e)}\`)` — captured by `wrangler tail`. |

### Documented incomplete-coverage gap (per plan §5.3)

When the facet path takes a **transitive 'fail' reject**, the facet throws via the `__w6_reject`-tagged Error. The supervisor's existing catch block at `npm-installer.ts:580–584` discards the original error and creates a fresh `Error("resolver-facet failed: …")` (this is the MF-1 from review). The events the facet collected before the throw are also lost because `result` was never assigned.

We could fix this by extending `pool.submit`'s `ExecutionError` to preserve own-properties across the supervisor↔facet boundary. The fix is more invasive than W6.5's scope (touches `parallel/vendor/errors.ts` plus the supervisor catch). Documented in `src/npm-resolve-facet.ts` comments at the relevant `__EMIT_EVENT` call site.

**What this loses in practice:**
- Facet-side transitive reject events are NOT emitted to the sink today. The user-visible install error (which contains the package name and reason) IS still produced — the tagged error survives at the user-error level via the `messages: string[]` channel that was always there.
- For "prioritize next swaps by demand" purposes, this means: top-level rejects (the most common signal — user typed `npm install <bad-package>`) are 100% captured, transitive rejects via supervisor BFS are 100% captured, transitive rejects via facet path are 0% captured. Facet path is the default in prod (W6 retro S1), so the gap matters but doesn't make the hook useless — top-level demand signal alone is the most actionable for swap-priority decisions.

W6.6 candidate: extend `ExecutionError` to preserve own-properties; emit facet-throw rejects.

### What this enables

`wrangler tail | grep '\[w6.5/registry\]'` for ~24h yields a JSONL stream. Aggregate `reject.from` frequencies; the top-N is the next swap-priority list. Cheap, no new infra, ships today. The real `analytics_engine_datasets` upgrade is deferred to F-observability (drafted in `audit/_drafts/F-observability.md`).

## 4. What surprised

### S1 — All five "swap candidates" turned out NOT to be swap candidates.

The W6.5 task spec listed 5 packages to add as WASM swaps:
- `sql.js` — actually a loader fix (not a swap; same require name).
- `@libsql/client` — actually a resolver fix (not a swap).
- `@swc/wasm-web` — actually a runtime-loader fix (not a swap; same require name).
- `sharp-wasm32` — actually a hard install-time block (`cpu: ["wasm32"]`) followed by a hard runtime block (libvips threads).
- `@napi-rs/canvas-wasm32-wasi` — does not exist on npm.

W6.5's honest delivery is 0 new SWAP entries. Track 1 (registry expansion) became registry refinement + telemetry + new REJECT entries. Track 2 (loader fixes) was entirely deferred. The plan v2 anticipated this risk in §12 ("counting only true SWAP entries, this likely ships 0 unless a Track 2 spike pays off") and v2 §13 ("zero new SWAPs disclosure") — making the call early, before implementation, prevented the W6 v1 overscope problem.

This is the W6 retro lesson exactly: the marketing-friendly "registry of swaps" headline is a small fraction of the actual value. The reject list, with verified-or-caveated alternatives, is the bigger artifact. W6.5 doubles down on that: 24 entries got refined `suggest:` text with honesty discipline (alts that fail in their `.out.txt` MUST be caveated), 3 new REJECTs catch additional footguns.

### S2 — `@napi-rs/canvas-wasm32-wasi` does not exist.

The W6.5 spec asked us to "verify viable" for sharp-wasm32 but listed `@napi-rs/canvas-wasm32-wasi` as a swap candidate without "verify viable" qualification. A `curl -sI https://registry.npmjs.org/@napi-rs%2Fcanvas-wasm32-wasi` returns 404. The `@napi-rs/canvas` project ships native-only.

The honest move was: add a REJECT entry for the non-existent package (so users typing `npm install @napi-rs/canvas-wasm32-wasi` get a useful error pointing at canvaskit-wasm) AND add a REJECT entry for the parent `@napi-rs/canvas` (so users who try the native one get the same pointer). W6.5's value here is preventing two futile install attempts.

### S3 — Spike outcomes flipped 2 of W6's "tracked-for-W6.5" promises.

W6 retro §5 said:
- "WASM extraction filter for `dist/*.wasm`. Unblocks `sql.js` and removes its REJECT entry. … Estimated ~30 LOC + e2e probe."
- "VFS pre-bundle wiring for `@swc/wasm-web`. Unblocks `@swc/wasm-web` and removes its REJECT entry. Estimated medium effort."

Both estimates were wrong. The actual gaps are:
- sql.js: not in tar extraction (no filter exists). The `dist/sql-wasm.wasm` file extracts fine; the runtime ENOENT is downstream — between the supervisor's writeBatch RPC handler and the runtime's fs.readFileSync VFS shim. Multi-file fix.
- @swc/wasm-web: not in pre-bundle slice walker (it walks every file). The `'Code generation from strings disallowed'` error fires from `node-shims.ts:2058`'s `new Function(code)` — a workerd CSP-like restriction. General facet-runtime issue, not registry scope.

Lesson: W6 retro's "tracked for W6.5" estimates were based on the simplest plausible diagnosis. W6.5's spike phase forced those diagnoses to be tested and corrected. The registry's REJECT `suggest:` text has been updated to be honest about "tracked for W6.5.x" with the ACTUAL scope (multi-file fix, general runtime issue) so the next wave starts from correct hypotheses.

### S4 — `_SUMMARY.json` was lying about 7 packages.

W6 retro §5 already flagged this ("S5 — `audit/probes/wasm/_SUMMARY.json` is misleading"). W6.5's `regression/summary-honesty.mjs` probe pinned the lie — every `<name>.out.txt` file is examined for known-fail markers, and the summary's `ok:true` is checked against actual load success. 7 of 12 entries flipped to `ok:false`. The probe is now a regression gate against future drift. The `note:` field gives one-line explanations of WHY load failed (verbatim from the .out.txt error).

### S5 — The plan-review caught a P0 telemetry design defect.

Plan v1 designed the telemetry hook to detect facet rejects via the supervisor's BFS catch (matching `__w6_reject` own-property). Review MF-1 found this is **factually wrong**: `npm-installer.ts:580–584` wraps the error and drops the tag. By the time supervisor code sees the failure, `__w6_reject` is gone. Plan v1's "default-prod facet path is captured" claim was the inverse of reality — facet-path rejects would be 0% captured.

Plan v2 redesigned to pipe events back through `ResolveFacetResult.registryEvents`. This works for the **success-path** (swap + transitive-skip events) but loses the throw-path (facet's transitive `fail` reject) — same W6.5.x deferred work as S1/S3. The retro is honest about the gap; the plan v1 design would have shipped a hook that emits ~5% of relevant events while claiming 100%.

Without the sub-agent review, this defect ships. The review investment (one sub-agent invocation, ~5 min) saved a re-do wave.
