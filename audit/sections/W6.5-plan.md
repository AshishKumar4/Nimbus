## 5. Telemetry hook design (Track 1) — REDESIGNED in v2 after MF-1

### 5.1 Requirement (from spec)

> When REJECT_INSTALL fires, emit a telemetry event so we can prioritize next swaps by demand.

### 5.2 Where rejects fire

Three call sites, per the W6 explore-report:

| Site | File | Function | Reject path |
|---|---|---|---|
| Top-level | `src/npm-installer.ts` | `applyW6Registry` (line 955) | Throws `RegistryRejectError` (typed). |
| Supervisor BFS (legacy resolver) | `src/npm-resolver.ts` | inside `resolveTree` (~line 547–565) | Throws `RegistryRejectError`. |
| Facet BFS (default prod path) | `src/npm-resolve-facet.ts` | `resolveOne` (~line 433–442) | Throws `Error` tagged `__w6_reject = true`. |

The default prod path is the facet path (`shouldUseFacetResolver()` returns true unless explicitly disabled, per W6 retro S1). **Telemetry that doesn't capture the facet path captures almost nothing real.**

### 5.3 MF-1 fix — ResolveFacetResult carries events

The plan v1 design assumed the supervisor's BFS catch-block could detect facet rejects via `__w6_reject` and emit there. Review found this is **factually wrong**: `src/npm-installer.ts:580–583` discards the original error and creates a fresh `new Error("resolver-facet failed: " + msg)`. By the time supervisor code sees the failure, `__w6_reject` is gone and `isRegistryReject(e)` returns false.

**Fix shape:** extend the existing `ResolveFacetResult` interface (declared at `src/npm-resolve-facet.ts:104`, mutated in `messages: string[]`) with an additional `registryEvents: RegistryEvent[]` field. The facet collects events into this array as it runs (push into a closure-bound array in `resolveTreeInFacet`); on success or failure, the array is returned. Supervisor reads it and flushes to the sink.

But the facet path also has a *throw* case (transitive `'fail'` reject). Two options:

**(a)** The facet pre-records the event into the array, then throws. The supervisor's catch wraps the error but the array is lost (because `result` was never assigned).

**(b)** The facet pre-records the event into a side-channel that survives both the success path and the throw path. The cleanest version: serialize the event as a string into the thrown Error's `message` (with a sigil prefix like `[w6.5/event]<json>`), and the supervisor parses these out of the failure message. Brittle.

**(c) Chosen:** combine both. The facet writes events into a `pendingEvents: RegistryEvent[]` array. On success, it returns them in `ResolveFacetResult.registryEvents`. On throw, it throws a custom `RegistryFacetError` (a plain `class extends Error` with `__w6_facet_events: RegistryEvent[]` own-property). The supervisor's catch in `npm-installer.ts:580–584` unwraps the error and reads `e.__w6_facet_events` (own-property — survives `pool.submit`'s `ExecutionError` wrap as long as we extend that wrap, which we will), emits each event, then re-throws as today.

Since `pool.submit`'s `ExecutionError` already does `JSON.parse`/`stringify` on remote errors (per W6 retro S4), we'll preserve own-properties by serializing/restoring them across the boundary. Alternative simpler path: have the supervisor's catch parse a known sigil from `e.remoteMessage` — strictly worse than option (c).

For Phase B/C I'll start with the most-conservative version: facet returns events on success (covers the `swap` and `transitive-skip` cases — far more frequent than `reject`). The facet-throw `reject` event is a known incomplete-coverage gap until ExecutionError-own-property-preservation lands. Compared to v1, this is a strict improvement: v1 lost ALL facet events; v2 captures all swap + transitive-skip events from facet, plus all reject + swap + transitive-skip events from supervisor BFS.

Document this gap explicitly in retro §F.

### 5.4 Sink API

Registry-side (`src/wasm-swap-registry.ts` — supervisor isolate only):

```ts
export type RegistryEvent =
  | { type: 'swap'; from: string; to: string; ctx: 'top' | 'transitive' }
  | { type: 'reject'; from: string; reason: string; suggest?: string; ctx: 'top' | 'transitive' }
  | { type: 'transitive-skip'; from: string; reason: string };

export type RegistryEventSink = (e: RegistryEvent) => void;

let _sink: RegistryEventSink | null = null;
export function setRegistryEventSink(s: RegistryEventSink | null): void { _sink = s; }
export function emitRegistryEvent(e: RegistryEvent): void {
  if (!_sink) return;
  try { _sink(e); } catch { /* must never break install */ }
}
```

The try/catch swallow (review MF-2 raised this) is intentional: telemetry must never break the exit path. To address the "hides bugs" concern, also export:

```ts
export function getRegistryEventSink(): RegistryEventSink | null { return _sink; }
// Test-only: when the probe wants to assert that the sink threw at least once.
let _sinkThrowCount = 0;
export function getSinkThrowCount(): number { return _sinkThrowCount; }
```

`emitRegistryEvent` increments `_sinkThrowCount` on caught error so production can monitor via `/api/_diag/...` if needed and probes can pin "did the sink throw?".

### 5.5 Isolate-safety invariant (MF-2)

The registry module is a per-isolate singleton. Two contexts to consider:

- **Supervisor isolate** (DO instance): one `_sink` per DO instance. Two concurrent installs on one DO would share, but `installLock` enforces serialization, so the per-install captured-events use case is uncontended.
- **Facet isolate** (`NimbusFacetPool`): the facet preamble does NOT import the registry. Therefore the facet has no `_sink` — events are collected into the array as designed in §5.3 instead of going through a sink at all. **No facet-side `setRegistryEventSink` exists.**
- **Probe isolation:** W6 `run-all.mjs` spawns each probe as a child `bun` process (line 48 `spawnSync('bun', [probe], …)`) — fresh isolate per probe. Per-probe sink injection is uncontended.

A `functional/sink-isolation.mjs` probe (added per MF-2) spawns two child `bun` processes concurrently, each setting a different sink and emitting events, asserts no cross-contamination via stdout markers.

In `src/index.ts` (worker entry, NOT the DO entry — `console.log` from the fetch handler reaches `wrangler tail` directly):

## 6. Transitive-WASM policy regression (Track 1) — RESCOPED in v2 after MF-3

W6 already implements transitive swap at three sites: supervisor `buildSpecs`, supervisor BFS, facet BFS. What's missing is a regression that **verifies the policy on a real tree shape** — proven by review MF-3 to be impossible without facet/cache scaffolding via the in-memory unit-test path the v1 plan implied.

**v2 decision (per MF-3):** ship two probes, each honest about what it covers.

### 6.1 `regression/transitive-swap-decision-rule.mjs` (network-free, in-memory)

A unit-style probe: builds a synthetic resolved tree as a plain JS object (e.g. `{ vite: { dependencies: { esbuild: '*', rollup: '*' } } }`), walks it with `applySwaps` per-spec via the registry's pure functions, asserts the post-walk shape replaces `esbuild` with `esbuild-wasm`. This is essentially a recursive `applySwaps` over a tree fixture — proves the *policy* is encoded correctly in the registry, doesn't prove the *resolver* applies it. Cheap, fast, deterministic.

### 6.2 `regression/resolver-paths-symmetric-w6.5.mjs` (extends W6's symmetric probe)

W6 has `audit/probes/w6/regression/resolver-paths-symmetric.mjs` which extracts decision rules from both supervisor and facet resolver source files and asserts they make the same per-package decision. Extend it with a new fixture row: a transitive-swap case (esbuild appearing as transitive dep).

### 6.3 `e2e/transitive-swap-prod.mjs` (PROD-GATED, OPTIONAL)

If `NIMBUS_W6_5_E2E_PROD=1`, drive a real prod session: `npm install vite` (vite has esbuild as a dep), then `ls /home/user/app/node_modules` and assert `esbuild-wasm` exists and `esbuild` does NOT. Default-skipped (since wrangler auth may be unavailable per W6 retro §7); same gate as `audit/probes/w6/e2e/registry-coverage.mjs`.

This trio (decision rule + symmetric extraction + optional prod e2e) is honest: the first two prove the wiring at code level, the third proves it at runtime when prod auth is available. Replaces v1's overstated "actual tree" claim.
