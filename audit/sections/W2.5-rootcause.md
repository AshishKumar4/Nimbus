# W2.5 Root Cause ‚Äî Bundle Truncation, Not Install Pipeline

> **Status:** Build-mode investigation 2026-04-30 against prod ver
> `edc98e62` (final fix at HEAD `<this commit>`). Root cause **identified
> and fixed**; W2.5a's defensive Set-add patch was a no-op (the SqliteVFS
> children-index was correct all along).
>
> **Verdict:** The bug was in the FACET BUNDLER (`buildVfsBundle` in
> `src/facet-manager.ts:453`), not in the install pipeline. SQL stored
> all 1928 inodes correctly. SqliteVFS.readdir returned all 46 packages
> correctly. The 500-file / 4 MiB content-bundle cap silently truncated
> the snapshot embedded in dynamic-facet workers, hiding ~37 of 46
> packages from `fs.readdirSync` inside user code.

---

## 1. What W2.5a actually did

The W2.5a patch (697040f) lifted `_addToChildrenIndex` out of the
`prior === undefined` guard in `src/sqlite-vfs.ts:1162-1216`, ensuring
the in-memory children index always reflects every writeBatch entry.
Diagnostic logging behind `NIMBUS_DIAG_INSTALL_PIPELINE=1` proved this
was a defensive no-op: across 191 logged events for an `npm install
fastify` on prod, **every** `stale-prior` event reported
`indexedBefore=true` (i.e. the children Set already contained the entry
before the patch ran), and **zero** `readdir size mismatch` events
fired. SqliteVFS was correct pre-W2.5a and post-W2.5a.

Concrete diag captures:
[`audit/probes/w25-diag-fastify.txt`](../probes/w25-diag-fastify.txt)
and
[`audit/probes/w25-diag-fastify2.txt`](../probes/w25-diag-fastify2.txt).

| Diag bucket | Count | Conclusion |
|---|---|---|
| `readdir miss` (kids undefined) | 1, on irrelevant `usr/lib/node_modules` | not pkg-related |
| `readdir size mismatch` (kids.size != results.length) | 0 | inodes map ‚Üî children index agree |
| `stale-prior` with `indexedBefore=true` | 190 | every "stale" already correctly indexed |
| `stale-prior` with `indexedBefore=false` | 0 | zero broken-index events |

W2.5a's "always re-affirm" patch is now redundant. We left it in place
(it's harmless ‚Äî `Set.add` is O(1) and idempotent) but it can be safely
reverted in a future cleanup wave.

## 2. The actual bug

`fs.readdirSync` and `fs.statSync` inside Node code running in a Nimbus
*facet* do **NOT** call `SqliteVFS.readdir`. The facet has no synchronous
RPC channel back to the supervisor DO, so it answers `readdirSync` from
a **pre-built snapshot** embedded in the facet's worker module:

```
src/node-shims.ts:256       readdirSync()  ‚Üê scans __vfsBundle keys
src/node-shims.ts:226       statSync()     ‚Üê scans __vfsBundle keys
src/facet-manager.ts:453    buildVfsBundle  ‚Üê supervisor-side snapshot builder
src/facet-manager.ts:185    embeds JSON.stringify(bundle) into the worker module
```

`buildVfsBundle` walked `node_modules` and short-circuited at **500
files / 4 MiB / depth 5** ([`src/constants.ts:19-21`](../../src/constants.ts)).
For an `npm install fastify` (1928 files / ~5 MiB extracted), this
truncated mid-iteration. The packages whose files happened to land before
the cap (alphabetic-ish but really insertion-order-of-`children`-Set,
which tracks `writeBatch` completion order) became visible; everything
else returned `ENOENT` from inside the facet.

Smoking-gun probe inside facet on the broken build (W2.5a deploy):

```
$ ls /home/user/app/node_modules        ‚Üê shell ls (calls SqliteVFS)
@fastify  @pinojs  abstract-logging  ajv  ajv-formats  atomic-sleep
avvio  cookie  dequal  fast-decode-uri-component  fast-deep-equal
fast-json-stringify  fast-querystring  fast-uri  fastify  fastq  ‚Ä¶  toad-cache
                                        ‚Üê 41 packages visible

$ node -e 'require("fs").statSync("/home/user/app/node_modules/fastq")'
ENOENT                                  ‚Üê Node fs (reads __vfsBundle snapshot)

VISIBLE files=485 dirs=62               ‚Üê walked from facet via fs.readdirSync
                                        ‚Üê 485 ‚âà 500-file cap; 25% of installed
```

The 1928 files were in SQL. The 41 package dirs were in
`SqliteVFS.children`. `ls` saw them. `node` did not. The W2 audit's
"install-empty" verdict for fastify et al. was a measurement artifact:
the install was fine; the bundle snapshot the smoke-test saw was not.

## 3. The fix

Two changes, both in `src/facet-manager.ts` + `src/node-shims.ts`:

### 3.1 Decouple directory shape from content cap (commit body)

`buildVfsBundle` now produces TWO outputs
([`src/facet-manager.ts:453-518`](../../src/facet-manager.ts), see
new `FacetVfsState` interface):

- **`bundle: Record<string, string>`** ‚Äî path ‚Üí file content, still
  capped (`VFS_BUNDLE_MAX_FILES`/`_BYTES`/`_DEPTH` in `src/constants.ts:19-21`).
  This is what feeds `require()` content reads.
- **`manifest: Record<string, string[]>`** ‚Äî path ‚Üí child-name list.
  Walks the same roots WITHOUT the cap (only a guard depth limit). For
  fastify's install this is ~50 KiB of JSON ‚Äî three orders of magnitude
  smaller than the 4-MiB content bundle.

Both are embedded into the dynamic-facet worker module (alongside each
other) so the facet has full directory-shape visibility regardless of
content truncation.

### 3.2 Manifest-aware fs shims

`src/node-shims.ts` `readdirSync`/`statSync`/`existsSync` consult
`__vfsManifest` first, falling back to the legacy `__vfsBundle`
prefix-scan only when the manifest doesn't cover the path (e.g. dirs
created at exec time via `mkdirSync`). New code paths:

- `readdirSync` (line 256) ‚Äî unions manifest children + bundle prefix
  matches + `__vfsWrites` + `__vfsDirs`.
- `statSync` (line 226) ‚Äî manifest entry ‚Üí directory; manifest sibling
  list ‚Üí file (zero-size; content lookup may still fail, but `stat`
  succeeds, which is what most tooling actually needs).
- `existsSync` (line 211) ‚Äî manifest membership test.

### 3.3 Priority pass for `package.json`

`buildVfsBundle` now does a priority pass over `node_modules` BEFORE
the body fill ([`src/facet-manager.ts:497-548`](../../src/facet-manager.ts)).
Each package's `package.json` is added with `force: true` so it bypasses
the byte cap (cumulative impact bounded ‚Äî ~46 √ó ~2 KiB ‚âà 100 KiB).
Main-entry candidates (`main` / `module` / `exports['.']`) are added
WITHOUT force ‚Äî they're best-effort; if they fit in the body pass, they
land.

This guarantees the require resolver can always read every installed
package's `package.json` to determine its main entry, even if the main
file's content itself didn't fit in the bundle. (When the main itself
also fits, `require(<pkg>)` works; when it doesn't, only that single
package's runtime smoke fails ‚Äî directory shape and resolution are
still correct.)

### 3.4 Diag retained, gated off

The W2.5b diagnostic logging in `src/sqlite-vfs.ts:907-947, 1162-1216`
remains in the source (cheap, behind a single-string env-var check) so
future investigations can re-enable via `wrangler.jsonc` `vars`
without a code change. `NIMBUS_DIAG_INSTALL_PIPELINE=""` is the default
(off).

## 5. Why the 33-package smoke-test count didn't move

The audit predicted +8 packages would flip ‚öÝÔ∏è‚Üí‚úÖ post-W2.5. Measured
delta is +0 (still 4 ‚úÖ: `better-sqlite3`, `jest`, `pg`, `zod`).
W2.5b correctly fixed `fs.readdirSync` / `fs.statSync` semantics, but
the W2 prediction conflated two distinct failure modes:

1. **Directory invisibility** ‚Äî pre-fix, packages weren't visible at
   all from the facet, so the resolver couldn't even start. **Fixed**
   by the manifest.
2. **Content unavailability** ‚Äî even now, with the package visible and
   `package.json` readable, the resolver may resolve the require to
   a file whose content was capped out of the bundle (`avvio/lib/index.js`
   for fastify, `pg-core/columns/enum.cjs` for drizzle-orm, etc.). The
   resolver then throws `Cannot find module` or `file was not pre-bundled`.

For the `+8 packages` target, the architectural fix is to make
`__readFileOr` (`src/node-shims.ts:921`) lazily fetch missing content
from the supervisor ‚Äî but `readFileSync` is synchronous and facet‚Üí
supervisor RPC is async-only in workerd. Options for a future wave:

- **(a)** Move the snapshot from "embed-in-worker-module" to
  "DO-storage-fetched-at-facet-boot" ‚Äî bypasses workerd's per-module
  size budget.
- **(b)** Move user code from sync facets to a worker that has sync
  storage access (RPC-allowed Durable Object class running inside
  the supervisor itself ‚Äî a different architecture).
- **(c)** Pre-bundle every package's full content via supervisor work
  before the facet boots, then stream into the facet's `__vfsBundle`
  via a one-time async `init()` call BEFORE running user code. This
  avoids the embed-in-module limit while keeping the rest of the shim
  synchronous.

These are W2.6 / W3-class architectural changes. Out of scope for the
2-hour W2.5b window. The minimal fix shipped here is sufficient
to unblock `readdirSync`-driven tooling (Vite scanner, esbuild's
plugin-resolve, postinstall scripts) which previously saw a phantom
node_modules.

## 6. Lessons

1. **Trust the SQL counter, not the smoke output.** The W2.5-plan's H5
   verdict ("symptom confirmed, mechanism unknown") was right to keep
   investigating after the conservative fix. Believing the "+8 ‚úÖ"
   prediction would have shipped a no-op patch under "looks fixed" cover.
2. **`fs.readdirSync` in a facet is not the SqliteVFS readdir.** This
   was the cognitive miss in the W2.5-plan: the audit traced symptoms
   to `SqliteVFS.children` divergence without realising user-code
   reads come from a DIFFERENT data structure (the embedded snapshot).
   The next investigation that sees a similar "SQL has it but user code
   doesn't" pattern should immediately suspect the bundler.
3. **Caps that silently truncate are footguns.** `buildVfsBundle`'s
   500-file cap had no observability ‚Äî no log, no error, no marker in
   the bundle that "more was elided here". The fix's manifest is
   uncapped specifically so directory shape is always honest, even when
   content isn't.
4. **W2.5a was a textbook "ship the conservative fix" mistake.** The
   plan correctly identified `prior !== undefined` as suspicious but
   committed to a defensive patch without diagnostic data to confirm.
   The patch landed, didn't fix anything, and burned a deploy slot.
   Future plans: the diagnostic step MUST run before the conservative
   step, even if it means a second deploy.

**Ready, with caveats.** W3's plan (`vm` / `http2` / `repl` builtin
shims for ~3 more packages) is independent of the bundle-cap issue.
W3 will turn ‚öÝÔ∏è ‚Üí ‚úÖ for axios (needs `http2`) and similar packages
where the failure is missing-builtin, not missing-content.

What W3 should add to its plan: package list filtered for "blocked
on builtin only, not on content cap" ‚Äî verify by re-running
`audit/probes/run-packages-prod-w2.mjs` and grepping for `Cannot find

`Cannot find module '<some-transitive-pkg>'` need the W2.6 architectural
fix (above), not W3.
