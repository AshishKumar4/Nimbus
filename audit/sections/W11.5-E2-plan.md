## TL;DR

`next dev` fails on Nimbus today because Next 14.2's webpack-based
dev server spawns a multi-layered worker pool through
`child_process.fork(... stdio:[…,'ipc'])` and our W8 substrate ships
JSON-IPC + a no-op `worker_threads` stub.  The failure ordering is
**S6 → S7 → S8 → S10** (per `audit/probes/w115-e2-investigation/R0-static-failure-projection.mjs`),
where S6 is W11.5-E1's ipc-shape gate and S10 is W9.5's hibernation
gate.  This document plans the *substrate* fix for **S7 + partial S8**,
codified as gate **E2 — webpack-in-facet**.

We pick **Option A (coalescing pool)**: detect webpack's jest-worker
client-side at fork-time and redirect its workers to a single,
long-lived **build-worker facet** that runs webpack *serially* over
each chunk request.  This reuses Nimbus's W8 facet substrate without
introducing new RPC contracts, doesn't depend on E1, and produces
deterministic build output that downstream gates (E1 ipc-shape,
W9.5 hibernation) can exercise independently.

Acceptance bar: a synthetic `webpack({ entry, output })` call from
within a Nimbus user shell completes `compiler.run()` and writes a
deterministic `dist/` to the VFS.  `next dev` end-to-end is the
*combined* gate of E1+E2+W9.5 — explicitly out of scope for E2 alone.

## 0. Wave context and gate decomposition

### 0.1 The two Phase-2 substrate gates (W11.5-E1, E2)

W11-retro §6 item 5 originally listed three Phase-2 substrate gates
for Next.js dev mode. The third (E3 — Cloudchamber container-in-DO
via SHIP-10537) has been **removed from the Nimbus roadmap**:
Nimbus is DO-only emulation, and Cloudchamber container-in-DO is
exactly the platform substrate Nimbus is designed to emulate
without. We track it as a primitive that exists; we do not depend
on it.

The remaining two gates are *independently dispatchable*, sequenced
loosely by which failure mode fires first when `next dev` runs
end-to-end:

| Gate    | Surface                                       | Status today                            | Owner    |
|---------|-----------------------------------------------|-----------------------------------------|----------|
| **E1**  | v8-serializer fork IPC                        | Platform-gated on workerd `node:v8`     | (E1)     |
| **E2**  | webpack-in-facet                              | **THIS PLAN** — non-platform-gated      | (this)   |

E1 owns the ipc-frame format mismatch.  Today our `child_process.fork()`
shim ships a **JSON-newline** projection (`src/node-shims.ts:1556-1648`)
because workerd's `node:v8` did not expose `serialize`/`deserialize`
when W8 shipped (W8-retro §3 item 2).  E1's plan will need to
re-investigate workerd's current `node:v8` surface; if available, route
parent.send/process.send through it; if not, document the gap deeper
in node-shims and continue.  **E1 is platform-gated** in the same
sense W7.5 was: it depends on a workerd capability we do not control.

E2 — this plan — owns the webpack-pool concurrency surface.  It is
**non-platform-gated** because every part of the fix lives inside
Nimbus's existing W8 + facet-pool code paths.  We can ship it without
waiting on workerd.

### 0.2 What "webpack's recursive worker-pool problem" means precisely

The brief frames the issue as *"webpack's recursive worker-pool
problem on Nimbus's facet-pool architecture."*  Concretely this is the
junction of three independent mechanisms colliding:

1. **Webpack 5's `parallelism` option** (default 100).  Each
   `Compilation` opens a `parallelism` queue against
   `compilation.processModuleDependencies`.  Inside Nimbus, that's
   100 microtask-promise concurrency slots — fine on its own.
2. **Plugin-driven worker pools.**  `terser-webpack-plugin` (default
   in `webpack.config.optimization.minimizer`) and
   `css-minimizer-webpack-plugin` use **`jest-worker`** under the
   hood.  jest-worker spawns N child processes via
   `child_process.fork({ stdio:[…,'ipc'] })` where N defaults to
   `os.cpus().length - 1`.  These are **real OS forks** in Node; on
   Nimbus they route through `cpSpawn` → `FacetProcessManager.spawn`
   (`src/facet-process.ts:230`).
3. **Next 14.2's render-server worker.**  Next itself spawns one or
   two parent workers via `child_process.fork()` from
   `dist/server/lib/router-utils/setup-dev-bundler.js` (the
   render-server, the dev-watcher, the websocket bridge — the exact
   set varies between 14.0 and 14.2).  These then *each* run their
   own webpack compiler that *each* spawns terser+css-minimizer pools.

The "recursion" is thus **not** a stack-overflow.  It's a **fan-out
shape**: parent shell → next dev → render-server fork → webpack
parallelism queue → jest-worker.fork × 4 (terser) + jest-worker.fork ×
4 (css) — *per webpack pass*.  Next runs **three** webpack passes
(server, client, edge), so the worst case is **24 simultaneous
jest-worker children** at depth 4 of the cp tree.  Per
`R2-cp-recursion-budget.mjs`, the depth is *under* `CHILD_PROCESS_MAX_DEPTH`
but the count pressures both the per-session facets allocation budget
AND the supervisor's RPC concurrency, AND the W7.5/E1 ipc-shape gate
fails for every one of them in parallel.

### 0.3 What was historically tried (and learned)

W8 originally proposed (W8-plan §8.5 BLOCKER-2) that
`child_process.spawn` would mint a **child facet** per spawned command.
The implementation simplified to running `facet-direct` *inline in the
supervisor isolate* (W8-retro §3 item 7), trading real isolation for
correctness — that simplification is what makes today's `cp.fork(node
…)` work for husky/concurrently/cross-spawn but **not** for webpack:
those well-behaved tools execute in 100ms and emit small messages,
while webpack's terser fork sits with 200 MiB of compiled
ModuleGraph in memory for 30+ s.  Inline-in-supervisor is exactly
wrong for that workload.

So **E2's substrate change reverses W8-retro §3 item 7 for one
narrow case**: webpack's worker pool gets a real (or pooled) facet,
not an inline-in-supervisor execution.

### 0.4 Why now

Per MASTER-ROADMAP, the X.5 buckets restored webpack as a healthy
package (X.5-F retro: webpack ✅).  That means `npm install next`
puts a usable `next/dist/compiled/webpack` on disk; the install
pipeline is no longer the gate.  The substrate is.

## 2. Root-cause hypothesis ranking

We rank six hypotheses by *expected diagnostic ROI* (how much of the
failure stack each accounts for) and assign a verdict from the
investigation receipts.

### H1 — Facet count cap (~5-6 concurrent)

**Hypothesis:** Nimbus's facet substrate hits a hard cap at ~5-6
concurrent facets per session and webpack's pool exceeds it.

**Evidence pro:**
- `NimbusFacetPool` default concurrency = 4
  (`src/parallel/facet-pool.ts:233`).  This is the *user-facing* pool
  wrapper; it has explicit slot reuse and a 4-slot warm-isolate budget.
- W8's `FacetProcessManager.spawn` mints a **new** ctx.facets.get()
  entry per child PID — see `src/facet-manager.ts:1247`.  Each entry is
  a separate DO storage facet.

**Evidence con:**
- jest-worker does NOT route through NimbusFacetPool.  Each cp.fork()
  goes through `__supervisor.cpSpawn` → `FacetProcessManager.spawn`
  (`src/facet-process.ts:230`) → not the user-facing pool.  The
  `concurrency=4` is irrelevant on this code path.
- CF DO's facets cap per session is *much* higher than 4 (commonly
  cited at ~50-64 in CF-internal docs; not in code).  At worst we use
  ~17 concurrent facets (vite + real-vite + install pool + 12
  jest-worker children + supervisor) — well under the cap.
- The W8-retro §3 item 7 simplification means today's
  `facet-direct` runs *inline in the supervisor* (no extra facet
  hop).  Ironically, that means **today** the count cap is
  irrelevant — but it's the wrong substrate for the webpack workload
  (memory crowding, no isolation).

**Verdict: REJECTED as the headline gate.**
Facet-count is **not** the immediate failure.  However, the verdict
matters for the fix-architecture ranking: option B (warm pool)
naturally caps the count, while option A (coalescer) reduces it to 1
by construction.

**Probe to verify in build wave:**
`audit/probes/w115/regression/r-facet-count-bound.mjs` — instrument
ctx.facets.get to count concurrent live facets during a synthetic
webpack run; assert ≤ 6.  Asserting *bounded* count rather than a
specific value gives us a regression alarm without locking in a
particular bound.

### H2 — RPC nesting when child facet → SUPERVISOR → another facet

**Hypothesis:** When webpack's jest-worker child facet calls back into
SUPERVISOR (via `env.SUPERVISOR.writeFile` etc.) and the supervisor
then dispatches *another* facet (e.g. cpSpawn for a sub-worker), the
RPC stack overflows or deadlocks.

**Evidence pro:**
- W8-plan §8.5 BLOCKER-2 documented exactly this risk in the
  "facet-direct should mint a child facet that runs the command via
  FacetManager.execStream" path; the implementation deliberately
  avoided the recursion by inlining (W8-retro §3 item 7 — "no
  recursion deadlock by construction").
- The X.5-G/F retros document a related concern: stub-leak between
  parent and child facets (`facet-pool.ts:678` `dispose()` notes —
  "QueueState::ACTIVE fatal" if SUPERVISOR stubs aren't disposed).

**Evidence con:**
- The supervisor is single-threaded under workerd's event loop; an
  `await` from a cp child to SUPERVISOR.cpSpawn yields to the supervisor's
  microtask queue, which spawns another cp child whose own `await`
  yields back, etc.  This is depth-bounded (the cap = 8 in
  `facet-process.ts:191` plus NIMBUS_CP_DEPTH propagation at line 232).
- The static-projection probe R2 confirms max depth = 5 with
  headroom = 3.
- The actual deadlock vector (BLOCKER-2) was avoided in W8 by
  inlining the dispatch; today's substrate has *no* recursion
  surface.

**Verdict: REJECTED with a caveat.**
Today's substrate doesn't recurse, so H2 doesn't fire.  **But option
A (coalescer) re-introduces a single facet hop**, which means we must
re-verify H2 doesn't fire under the new architecture.

**Probe to verify in build wave:**
`audit/probes/w115/regression/r-rpc-no-deadlock.mjs` — synthetic
worker call chain (worker → SUPERVISOR.x → SUPERVISOR.spawn → other
worker → SUPERVISOR.y) with timeout assertion; should complete
within 5 s.

### H3 — jest-worker requires real fork() (W8 Phase 1 limit)

**Hypothesis:** jest-worker's WorkerPool depends on the *exact* Node
fork() semantics (copy-on-write heap, real file descriptors, real
IPC) that W8 Phase 1 does not provide.

**Evidence pro:**
- W8-retro §3 item 1 explicitly: "No real fork() copy-on-write
  semantics. fork() in this impl is `spawn('node', [modulePath, ...])`
  with an IPC channel multiplexed over stdin (parent→child) and
  stdout (child→parent). Real Node forks share initial heap with the
  parent; we mint a fresh isolate."
- jest-worker's INTERNAL contract (per
  `https://github.com/jestjs/jest/blob/main/packages/jest-worker/src/workers/ChildProcessWorker.ts`
  in 27/29.x) uses `process.send` and `process.on('message')` —
  exactly the v8-IPC channel.
- W8-retro §3 item 2: "fork IPC is JSON, not v8.serialize" — Buffer
  round-trips as `{type:'Buffer', data:[...]}`, Date as ISO string,
  Map/Set lose entries.  jest-worker's `args` arrays often contain
  Buffers (terser passes source code as Buffer).

**Evidence con:**
- **Heap CoW is not load-bearing.**  jest-worker doesn't depend on
  the child sharing the parent's heap — the child re-imports its
  worker module from scratch.  CoW is an *optimization* in real Node;
  Nimbus's fresh-isolate semantics are functionally equivalent.
- **stdin-queue IPC is functionally adequate** for the JSON envelope
  shape jest-worker uses, *modulo* the Buffer-projection issue.

**Verdict: PARTIAL ACCEPT.**
The "real fork" angle is a red herring; CoW doesn't matter.  But the
**ipc-shape** angle is real and is exactly W11.5-E1's gate.  E2 must
**not** depend on the Buffer-projection issue being fixed first; it
must produce a usable substrate even with JSON-only IPC.

**Probe to verify in build wave:**
`audit/probes/w115/functional/f-coalescer-no-ipc-deps.mjs` — drive
the build-worker facet directly with non-Buffer args (string +
JSON-only); assert it works without v8-IPC.

### H4 — webpack MessageChannel pattern vs our IPC bridge

**Hypothesis:** webpack itself uses `MessageChannel` (the WHATWG-style
postMessage channel from `node:worker_threads`) for its module
parallelism, not just `child_process.fork`, and our `worker_threads`
stub is a no-op (`src/node-shims.ts:1845`).

**Evidence pro:**
- webpack 5 has `Compilation.parallelism` defaulting to 100 (per
  `https://github.com/webpack/webpack/blob/v5.90.0/lib/Compilation.js`).
  This uses an internal `AsyncQueue` over Promise.all-style
  microtasks; **NOT** worker_threads.  Confirmed by reading webpack
  source: `Compilation.processModuleDependencies` is microtask-bound,
  not worker-bound.
- However, **plugins** can opt into worker_threads.  The
  `thread-loader` and `swc-loader` plugins do.  Next 14.2's default
  config does NOT use thread-loader (it uses swc-loader, which runs
  in-thread).  But terser-webpack-plugin's *internal* worker is
  jest-worker (cp.fork), not worker_threads.

**Evidence con:**
- Per R1, our `worker_threads` shim is a no-op `Worker` class
  (`new Worker(); .terminate(); .postMessage(); — all no-op`).  This
  is correct *only* if no consumer in the active stack actually
  postMessages and waits for a reply.
- Next 14.2's default webpack config does not use thread-loader.

**Verdict: REJECTED for Next 14.2's default path; ACCEPTED as a
forward-risk for Next 15+ / SWC `--experimental-build-worker`.**

The worker_threads stub *will* bite us when:
- The user installs `next@15+` which has experimental
  build-worker via worker_threads (per Vercel's blog posts).
- The user adds `thread-loader` to a webpack config.
- terser-webpack-plugin >= 6 (not yet shipped) might switch from
  jest-worker to worker_threads.

**E2's option A coalesces jest-worker; it does NOT directly fix
worker_threads.**  We address worker_threads via a *follow-up
extension* (option A.1, see §3.2.1 below) where we coalesce
worker_threads.Worker the same way as jest-worker.

**Probe to verify in build wave:**
`audit/probes/w115/regression/r-worker-threads-noop-still-noop.mjs` —
assert `worker_threads` shim shape unchanged post-E2 (so existing
W3 surface tests for `os.cpus()`-like consumers don't regress).

### H5 — terser/babel use worker_threads (partially shimmed)

**Hypothesis:** Inside each jest-worker child, terser hashing uses
`worker_threads.Worker` internally for parallel hashing of the chunk
output, and our partial shim hangs at postMessage.

**Evidence pro:**
- terser 5.x (`https://github.com/terser/terser`) does **not** use
  worker_threads internally; it's single-threaded in-process.
- HOWEVER, the **wrapper** terser-webpack-plugin 5.x has a
  `parallel` option that defaults to `os.cpus().length - 1` and uses
  jest-worker — already covered by S7.
- babel `@babel/parser` is single-threaded; no worker_threads.
- swc-loader (Next's default minifier in 14.2) runs in-thread.

**Evidence con:**
- The above contradicts the hypothesis as stated.  terser+babel
  themselves do NOT use worker_threads.  worker_threads pressure
  comes from *other* tools — H4 covered the `thread-loader` /
  `swc-loader --experimental-build-worker` cases.

**Verdict: REJECTED for the named tools (terser, babel) on Next
14.2's default config.**

The "partially shimmed worker_threads" remains a **forward-risk** for
options the user enables explicitly (thread-loader, build-worker), but
it is NOT the gate today.

**Probe to verify in build wave:**
`audit/probes/w115/regression/r-no-worker-threads-in-fixture.mjs` —
walk the webpack-minimal fixture's plugin list and assert no
thread-loader / worker_threads dependency.

### H6 — Hibernation mid-build kills child facets

**Hypothesis:** DO hibernation after ~30 s idle tears down all
in-flight cp child facets, leaving webpack with half-emitted chunks.

**Evidence pro:**
- W9 hibernation fires at the ws-keepalive-timeout boundary; default
  is the WS auto-response timeout configured in the W9 wave (see
  `src/nimbus-session-hib.ts`).
- A first-time webpack build of a non-trivial Next app takes 10-45 s.
  If the user types `npm run dev` and waits passively, the WS
  timeout can fire before the build emits.
- W9 hibernation tears down LOADER-loaded workers + facets at alarm
  time; in-flight cp-children would be terminated mid-RPC.

**Evidence con:**
- W9 was designed assuming user activity exists during long
  operations.  The *active* state during webpack build is
  `npm run dev` running — that itself is a long-running facet that
  should keep the supervisor warm.  But during webpack's compile
  phase, the supervisor is mostly idle (waiting on RPC from facets)
  and the WS may show no activity.
- Hibernation can be configured via `setHibernatableWebSocketEventTimeout`;
  Nimbus may already pin a longer timeout for active dev sessions
  (see W9-retro for tuning).

**Verdict: ACCEPTED as a forward-risk; OWNED BY W9.5, NOT E2.**

Hibernation mid-build is real, but it's a *separate* gate.  The fix
is in W9.5 (heartbeat from the build-worker facet to the supervisor).
E2's plan must NOT silently depend on hibernation tuning; E2's
acceptance gate must be reachable without W9.5 by running the
synthetic webpack call in a single supervisor-RPC turn.

**Probe to verify in build wave:**
`audit/probes/w115/e2e/e-build-completes-without-hibernation.mjs` —
synthetic webpack run with a 30-s ceiling on each step; explicit
heartbeat suppressed; assert build completes inside the WS-keepalive
window OR that the build-worker facet self-heartbeats.

| H  | Hypothesis                           | Verdict     | Owner of any residual fix    |
|----|--------------------------------------|-------------|------------------------------|
| H1 | Facet count cap                      | REJECTED    | guarded by E2 option A      |
| H2 | RPC nesting deadlock                 | REJECTED*   | re-verified by E2 build wave |
| H3 | jest-worker needs real fork()        | PARTIAL     | E1 (ipc-shape) — not E2     |

The *load-bearing* hypothesis for E2 is **none of H1-H6 in
isolation** — instead, it's the **interaction**: webpack's pool
shape multiplied by Nimbus's per-cp-child facet allocation creates a
12-fanout worst case that, even though under every individual cap,
multiplies fault-injection surface to the point that ANY of E1, H4,
H6 firing once kills the build.  The fix is to **collapse the
fanout** — option A.

## 3. Fix architecture options

We pick ONE option backed by the H1-H6 verdicts.  The four options
are sketched first, then ranked and chosen.

### 3.1 Option A — Coalescing build-worker facet (PROPOSED)

**Idea:** Detect that a `cp.fork()` is being called from inside a
webpack/jest-worker context — specifically, the parent module is
`jest-worker/build/index.js` or the env contains `JEST_WORKER_ID` or
the modulePath ends in `worker.js` AND args[0] is one of jest-worker's
worker entry filenames.  Redirect ALL of those forks to a single,
long-lived **build-worker facet** that runs each request serially.

**Architecture:**

```
                supervisor isolate (NimbusSession)
                ┌──────────────────────────────────────┐
                │  cpSpawn(req)                        │
                │   │                                  │
                │   ▼                                  │
                │  detectWebpackPool(req)              │
                │   ├── matches → coalesce()           │
                │   └── else    → existing dispatch    │
                │                                      │
                │  buildWorkerFacet (singleton)        │  ← long-lived
                │   - mints once per session            │     facet
                │   - serializes incoming requests via  │
                │     a queue                           │
                │   - per-request:                      │
                │       a. take next req from queue     │
                │       b. spawn webpack call inline    │
                │          inside facet via execStream  │
                │       c. emit stdout/stderr back via  │
                │          existing cp output ring      │
                │       d. mark child PID exit          │
                │   - shared isolate, shared            │
                │     compiled-modules cache            │
                └──────────────────────────────────────┘
                        │ (via ctx.facets.get/abort)
                        ▼
                build-worker facet (one isolate, many requests)
                ┌──────────────────────────────────────┐
                │  on each request:                    │
                │    require(workerScript)             │
                │    invoke worker[fnName](...args)    │
                │    return result                     │
                │                                      │
                │  shared cache of compiled scripts    │
                │  shared esbuild instance             │
                │  no fork — replaces all jest-worker  │
                │  children with serial dispatch       │
                └──────────────────────────────────────┘
```

**Code-touch surface (estimate):**

| File                            | Edit                                  | LOC est. |
|---------------------------------|---------------------------------------|---------:|
| `src/facet-process.ts`          | `_dispatch` → `detectWebpackPool` → coalesce branch | +80 |
| `src/build-worker-facet.ts` (NEW) | facet code template + singleton manager | +280 |
| `src/facet-manager.ts`          | `getBuildWorkerFacet()` lazy-init | +60 |
| `src/node-shims.ts`             | tag jest-worker forks with env hint at call time | +20 |
| `src/nimbus-session-helpers.ts` | export `JEST_WORKER_PARENT_FILES` constant | +15 |
| **Total**                       |                                       | **~455** |

**Why option A:**

- **Bounded count.**  By construction, all 12 jest-worker children
  collapse into one facet.  H1 verdict's "facet count" concern goes
  to zero.  Per-session live facet count drops back to ~5 (vite +
  real-vite + install pool + supervisor + build-worker).
- **Doesn't depend on E1.**  Coalescing happens *before* IPC;
  jest-worker's child code runs in a real Node-style require()
  context inside the build-worker facet without postMessage at all.
  The serialized request enters via plain RPC arg; the response
  returns via plain RPC return value.
- **Doesn't depend on W9.5.**  The build-worker facet is long-lived
  and explicitly heartbeats every 5 s to suppress hibernation.
- **Reverses W8-retro §3 item 7 narrowly.**  Inline-in-supervisor was
  the wrong substrate for webpack's heap pressure; the build-worker
  facet gives webpack its own ~128 MiB isolate.
- **Compatible with all five frameworks.**  Webpack runs the same
  way under Next, CRA, vite-legacy, vue-cli, etc.  The detection
  lives in a single regex over jest-worker entry-file names, so
  every consumer benefits.

**Risks:**

- **Detection accuracy.**  False positive (we coalesce a non-webpack
  fork) would silently break that fork's IPC.  False negative (we
  fail to coalesce a webpack fork) would leave the failure
  unchanged.  See §3.4 detection heuristic for mitigation.
- **Reintroduces RPC nesting (H2).**  The build-worker facet calls
  back to SUPERVISOR for VFS reads (the webpack input chunks).  The
  build wave's regression probe `r-rpc-no-deadlock.mjs` (§4.2)
  asserts no deadlock under high parallelism.

### 3.2 Option B — Pool reuse (warm-pool with explicit lifecycle)

**Idea:** Don't coalesce; rebuild jest-worker's pool semantics on
top of `NimbusFacetPool`.  Mint 4 stable-slot isolates at session
boot, route all jest-worker forks to those slots round-robin, with
explicit dispose() at session end.

**Code-touch surface (estimate):**

| File                          | Edit                                | LOC est. |
|-------------------------------|-------------------------------------|---------:|
| `src/parallel/build-pool.ts` (NEW) | NimbusFacetPool subclass with build-tagged slots | +220 |
| `src/facet-process.ts`        | `_dispatch` → BuildPool route       | +60 |
| `src/facet-manager.ts`        | wire shared BuildPool                | +40 |
| `src/node-shims.ts`           | as for option A                      | +20 |
| **Total**                     |                                     | **~340** |

**Why NOT option B:**

- **Doesn't fix H6.**  Hibernation tears down each pool slot
  individually; we'd need 4× the heartbeat surface.
- **Doesn't fix forward-risk H4.**  worker_threads stays no-op.
- **Larger blast radius on facet-pool.ts.**  facet-pool is the
  install path's hot loop (W4 W6 W8 X.5-F all depend on it).  Adding
  a build-tagged slot risks regression in install latency.
- **Saves no code over Option A.**  ~340 vs ~455 LOC, but the build
  pool ends up needing the same facet template anyway.

**Use this option if:**
Future profiling shows the single-facet-serial dispatch is too slow
for medium-to-large Next apps (build > 60 s).  Then option B is the
upgrade path.

### 3.2.1 Option A.1 — Option A + worker_threads coalesce (extension)

**Add-on to Option A:** Also intercept `worker_threads.Worker`
construction and route postMessage/onmessage frames through the
same build-worker facet (serializing requests into a JSON envelope
matching the WHATWG MessageChannel shape).

**Why optional:**
- Out of scope for Next 14.2's default config (per H4 / H5 verdicts).
- Forward-fits Next 15+ build-worker.
- Adds ~120 LOC to `src/node-shims.ts` (Worker class shim → cpSpawn
  envelope).

**Disposition:**
**DEFERRED to W11.5-E2.5.**  The build wave for E2 ships option A
only; A.1 is a follow-up if the user opts into Next 15 / thread-loader.

### 3.3 Option C — Synchronous bypass (in-supervisor serial shim)

**Idea:** Don't even allocate a build-worker facet.  Override
jest-worker's WorkerPool entirely at the `node-shims` layer with a
synchronous in-supervisor dispatcher that just `require()`s the
worker module and calls its export inline.

**Why NOT option C:**

- **Reverts to W8-retro §3 item 7's wrong substrate.**  Webpack's
  ModuleGraph in supervisor heap = OOM risk (W5 territory) under any
  non-trivial app.
- **No isolation.**  A bug in webpack/terser crashes the entire
  Nimbus session.
- **Concurrent installs share state.**  Two `next dev` runs in
  different terminals would share the same single-threaded webpack
  state.

**Use this option if:**
The build-worker facet itself has bugs at first ship and we need an
emergency fallback for users already on `next@14`.  Then option C
becomes a feature flag (`NIMBUS_NEXT_INLINE=1`).

### 3.4 Option D — Real fork() via Cloudchamber container — REJECTED

**Idea:** Replace W8's inline-supervisor execution with a real
Linux container via Cloudchamber container-in-DO.  Real fork(),
real worker_threads, real signals.

**Disposition: REJECTED.**  Nimbus is DO-only emulation;
Cloudchamber is exactly the platform substrate we are emulating
without.  Adopting Cloudchamber would defeat the project's
purpose.  We track Cloudchamber as a platform primitive that
exists ([cf-internal-dossier.md §6](../../docs/research/cf-internal-dossier.md))
but the Nimbus roadmap does not depend on it landing.

### 5.1 detectWebpackPool — `src/facet-process.ts`

Insert near `_dispatch` (`src/facet-process.ts:293`):

```ts
/**
 * Detect whether a cp.fork() request is being driven by jest-worker
 * (which webpack/Next use for terser, css-minimizer, thread-loader).
 *
 * Heuristic — three signals, any TWO of which trigger a match:
 *   1. modulePath ends in /jest-worker/build/workers/processChild.js
 *      OR /processChild.js OR /threadChild.js
 *   2. env contains JEST_WORKER_ID (jest-worker tags every spawn)
 *   3. parent stack frame mentions jest-worker (read from
 *      __supervisor's last RPC frame; cheap)
 *
 * False-positive cost: a husky-style fork that happens to ship a
 * `processChild.js` (none in the wild) would be mis-routed; the build
 * wave's f7 probe asserts no false positives in the W8 basket.
 *
 * False-negative cost: a future jest-worker rename would slip through;
 * the build wave's f1 probe pins the regex to today's surface.
 */
export function detectWebpackPool(req: SpawnReq): boolean {
  const path = req.command === 'node'
    ? (req.args[0] || '')
    : req.command;
  const sig1 = /jest-worker\/(build\/workers\/)?(processChild|threadChild)\.js$/.test(path)
    || /\/(processChild|threadChild)\.js$/.test(path);
  const sig2 = !!(req.env?.JEST_WORKER_ID);
  // sig3: cheap stack-hint check from the currently-recorded RPC
  // frame.  oom-discriminator.ts:getLastRpcFrame() is already
  // fail-safe.
  let sig3 = false;
  try {
    const frame = getLastRpcFrame();
    sig3 = frame?.includes('jest-worker') ?? false;
  } catch { /* fail-safe */ }
  // Two-of-three.
  return [sig1, sig2, sig3].filter(Boolean).length >= 2;
}
```

### 5.2 src/build-worker-facet.ts — NEW

Skeleton (~280 LOC):

```ts
/**
 * build-worker-facet.ts — singleton long-lived facet for webpack /
 * jest-worker workloads.  Coalesces what would otherwise be N
 * concurrent cp.fork() facets into a single isolate that processes
 * requests serially.
 *
 * Lifecycle: lazily created on first dispatchToBuildWorker call;
 * stays alive for the session; explicitly disposed on
 * NimbusSession.alarm() / kill-all.
 *
 * Heartbeat: emits a SUPERVISOR.heartbeat() every 5 s to suppress
 * hibernation while a build is in progress.  When the queue is
 * empty for >10 s, stops heartbeating — hibernation may then
 * legitimately fire.
 */

import { CF_COMPAT_DATE } from './constants.js';
import { getCtxExports } from './ctx-exports.js';

const BUILD_WORKER_FACET_NAME = 'build-worker';

const BUILD_WORKER_CODE = `
import { DurableObject } from "cloudflare:workers";
import * as __real_module from 'node:module';

const __compiledScripts = new Map();

function __resolveAndRequire(modulePath, supervisor) {
  // Read the script via SUPERVISOR.readFile, compile once, cache
  // by content hash.  Subsequent calls reuse the warm function.
  // Heap budget: webpack's worker is ~30 MiB resident; we cap the
  // cache at 16 entries.
  const cached = __compiledScripts.get(modulePath);
  if (cached) return cached;
  // (Pseudocode — full impl threads readFile through SUPERVISOR.)
  const src = supervisor.readFileSync(modulePath);
  const fn = new Function("module", "exports", "require", src);
  const mod = { exports: {} };
  fn(mod, mod.exports, /* require shim */);
  __compiledScripts.set(modulePath, mod.exports);
  if (__compiledScripts.size > 16) {
    const firstKey = __compiledScripts.keys().next().value;
    __compiledScripts.delete(firstKey);
  }
  return mod.exports;
}

export class BuildWorker extends DurableObject {
  async dispatch(reqJson) {
    const req = JSON.parse(reqJson);
    const sup = this.env?.SUPERVISOR || null;
    if (!sup) throw new Error('build-worker: no SUPERVISOR binding');
    const exports = __resolveAndRequire(req.modulePath, sup);
    // jest-worker entry contract: the module exports a default (or
    // .default) function, OR named functions matching args[0] (the
    // method name).  We try .default first, then args[0].
    const method = req.args && req.args[0];
    const fn = (method && exports[method]) || exports.default || exports;
    if (typeof fn !== 'function') {
      throw new Error('build-worker: module ' + req.modulePath +
        ' has no callable export (looking for ' + (method || 'default') + ')');
    }
    const result = await fn.apply(null, req.args.slice(1));
    return JSON.stringify({ exitCode: 0, result });
  }

  async heartbeat() { return true; }
}
`;

export class BuildWorkerFacetManager {
  private ctx: DurableObjectState;
  private env: any;
  private facetStub: any | null = null;
  private heartbeatTimer: any = null;
  private requestQueue: Array<() => Promise<void>> = [];
  private inflight = false;

  constructor(ctx: DurableObjectState, env: any) {
    this.ctx = ctx;
    this.env = env;
  }

  private async _ensure(): Promise<void> {
    if (this.facetStub) return;
    const ctxExports = getCtxExports();
    const supervisorBinding = ctxExports?.SupervisorRPC
      ? ctxExports.SupervisorRPC({ props: { doId: this.ctx.id.toString(), pid: 0 } })
      : undefined;

    const worker = this.env.LOADER.get(`build-worker:${this.ctx.id.toString()}`,
      async () => ({
        compatibilityDate: CF_COMPAT_DATE,
        compatibilityFlags: ['nodejs_compat', 'nodejs_compat_v2'],
        mainModule: 'worker.js',
        modules: { 'worker.js': BUILD_WORKER_CODE },
        ...(supervisorBinding ? { env: { SUPERVISOR: supervisorBinding } } : {}),
      }));

    const Cls = worker.getDurableObjectClass('BuildWorker');
    this.facetStub = (this.ctx as any).facets.get(BUILD_WORKER_FACET_NAME,
      async () => ({ class: Cls }));

    this._startHeartbeat();
  }

  private _startHeartbeat(): void {
    if (this.heartbeatTimer) return;
    this.heartbeatTimer = setInterval(async () => {
      try { await this.facetStub?.heartbeat(); } catch {}
    }, 5_000);
  }

  private _stopHeartbeat(): void {
    if (this.heartbeatTimer) { clearInterval(this.heartbeatTimer); this.heartbeatTimer = null; }
  }

  /**
   * Serial dispatch: enqueue, wait turn, dispatch, return.
   * The queue prevents two concurrent webpack passes from clobbering
   * the build-worker's compiled-modules cache.
   */
  async dispatch(reqJson: string): Promise<{ exitCode: number; result: any }> {
    await this._ensure();
    return new Promise((resolve, reject) => {
      const job = async () => {
        try {
          const out = await this.facetStub!.dispatch(reqJson);
          resolve(JSON.parse(out));
        } catch (e) { reject(e); }
      };
      this.requestQueue.push(job);
      this._drain();
    });
  }

  private async _drain(): Promise<void> {
    if (this.inflight) return;
    this.inflight = true;
    try {
      while (this.requestQueue.length > 0) {
        const job = this.requestQueue.shift()!;
        await job();
      }
    } finally {
      this.inflight = false;
      // Stop heartbeating after 10 s idle so hibernation can fire if
      // the user is genuinely idle.
      setTimeout(() => {
        if (this.requestQueue.length === 0 && !this.inflight) {
          this._stopHeartbeat();
        }
      }, 10_000);
    }
  }

  dispose(): void {
    this._stopHeartbeat();
    try { (this.ctx as any).facets?.delete(BUILD_WORKER_FACET_NAME); } catch {}
    this.facetStub = null;
  }
}
```

## 7. Open architectural decisions for user

These are **explicit calls** the user must make before the build wave
dispatches (or accept the proposed default).

### D1. Pool sizing — fixed vs adaptive

The proposed plan ships **option A** with a **single-singleton**
build-worker (effective parallelism = 1).  This trivially eliminates
the count concern but limits build throughput on multi-core
workstations.

| Choice | Pros | Cons |
|--------|------|------|
| **D1.a — Singleton (PROPOSED)** | trivially-bounded count, simple lifecycle, reuses one warm isolate (compiled-modules cache hit on every chunk) | wall-time = serial; 4-core build machines see no speedup |
| D1.b — Fixed N=4 | parity with jest-worker default; 4 × 30 MiB heap = 120 MiB still fits in workerd | 4 facets to manage; cache fragments |
| D1.c — Adaptive | size to `os.cpus()`; auto-shrink on heap pressure | complex; adaptive logic itself is a new failure surface |

**Recommendation: D1.a (singleton).**  Build throughput is **not**
the gate today — *correctness* is.  Singleton lets us ship E2 cleanly;
D1.b can be a follow-up (W11.5-E2.5) gated on observed wall-time
unhappiness.

### D2. Coalescer trigger heuristic — process-name match? RPC pattern? Marker env?

Per §5.1, the proposed detector uses **two-of-three** signals:

1. modulePath regex (jest-worker entry files)
2. JEST_WORKER_ID env present
3. RPC frame contains 'jest-worker'

| Choice | Pros | Cons |
|--------|------|------|
| **D2.a — two-of-three (PROPOSED)** | resilient to any one signal being missing/changing | slightly slower (3 regex/string checks per cpSpawn) |
| D2.b — process-name-only (modulePath regex) | simplest | breaks on Next bundling jest-worker into next/dist/compiled (already accommodated, but edge cases possible) |
| D2.c — env-marker only (JEST_WORKER_ID) | lowest false-positive rate | jest-worker in some configs doesn't set JEST_WORKER_ID until the child reads its env; we'd miss the spawn moment |
| D2.d — strict opt-in (require user to set NIMBUS_E2_COALESCE=1) | zero behavior change unless requested | webpack/Next users hit the bug by default and have to discover the flag |

**Recommendation: D2.a (two-of-three).**  Robust without forcing
user opt-in; f1 + f7 probes form the regression net.

### D3. Fallback if option A proves insufficient

Suppose the build wave ships A and the e2e probe e4
(`next-dev-fixture-build-completes`) still fails for reasons not
yet anticipated (e.g. webpack config-builder uses a non-jest-worker
parallelism path we missed).  What's the contingency?

| Choice | Pros | Cons |
|--------|------|------|
| **D3.a — Add option C as feature flag (PROPOSED)** | emergency fallback that bypasses jest-worker entirely; ships in same wave | revives the W8-retro §3 item 7 simplification; webpack heap pressure on supervisor |
| D3.b — Wait for E1 (ipc-shape) and re-test | E1 might unblock the secondary failure path | E1 is platform-gated; could be months |
| D3.c — Add option B (warm pool) on top of A | cleaner long-term architecture | larger code change; postpones any user-facing fix |

**Recommendation: D3.a (option C as feature flag).**  Ship A as
default; C as `NIMBUS_E2_INLINE=1` opt-in for emergency.  Tracks
toward W11.5-E2.5 if observed.

### D4. Heartbeat strategy

§5.2 proposes a 5-s heartbeat on the build-worker facet, stopping 10
s after queue idle.

| Choice | Pros | Cons |
|--------|------|------|
| **D4.a — 5s heartbeat (PROPOSED)** | well below WS auto-response timeout (typically 30s) | adds ~12 RPCs/min while build is active; small workerd overhead |
| D4.b — no heartbeat; rely on incoming work | simpler | hibernation may fire mid-build |
| D4.c — adaptive heartbeat (1 s during active compile, 30 s during link phase) | optimizes RPC overhead | complex |

**Recommendation: D4.a (5s).**  Simple, observable, reversible.

### D5. Build-worker facet's environment variables

The build-worker facet inherits `env.SUPERVISOR` for VFS reads.  Does
it ALSO inherit the user's full `process.env`?

| Choice | Pros | Cons |
|--------|------|------|
| **D5.a — Subset (PROPOSED)** | Pass NODE_ENV + NEXT_TELEMETRY_DISABLED + a curated whitelist | builds may need user-defined env vars; risk of "works on my session, fails on prod" |
| D5.b — Full inheritance | matches Node fork() semantics | leaks user secrets into a warm-cached isolate |
| D5.c — Strict empty | safest | breaks any build that reads env |

**Recommendation: D5.a (subset).**  Whitelist:
`NODE_ENV, NODE_OPTIONS, NEXT_TELEMETRY_DISABLED, NEXT_PHASE,
NEXT_RUNTIME, BUILD_ID, NIMBUS_FORK_IPC, NIMBUS_FORK_PARENT_HINT,
JEST_WORKER_ID`.  Anything else passes via the per-request envelope.

### D6. Should we eagerly initialize the build-worker facet on session boot, or lazy?

| Choice | Pros | Cons |
|--------|------|------|
| **D6.a — Lazy on first match (PROPOSED)** | zero cost for non-webpack sessions | first webpack request pays the facet-mint cost (~200ms) |
| D6.b — Eager at session boot | first webpack request is fast | every session pays ~200ms, even those that never run webpack |
| D6.c — Eager on `npm run dev` detection | targeted prewarm | requires hooking into the W11 framework detect path |

**Recommendation: D6.a (lazy).**  Most sessions are not webpack
workloads; lazy is the right default.

### D7. Should E2 lift the W11 next loud-block?

The W11 loud-block at `src/nimbus-session-init.ts:1352-1362` fires
when `npm run dev` runs against a Next package.  The user can bypass
with `--allow-next`.

| Choice | Pros | Cons |
|--------|------|------|
| **D7.a — Keep loud-block (PROPOSED)** | E2 alone doesn't fix Next end-to-end (E1 + W9.5 still needed) | users have to remember `--allow-next` to test |
| D7.b — Lift loud-block | better discoverability of the new substrate | misleading green for users; "works for build phase, dies on dev server" |
| D7.c — Make it a soft warning instead of a block | hybrid | similar to D7.b's downside |

**Recommendation: D7.a (keep loud-block).**  The deterministic
red-honesty is itself a deliverable (W11-retro §10 closing argument).
Lift the block when E1 and W9.5 also land.

### D8. Worker_threads (option A.1) — bundled in this wave or follow-up?

Per H4/H5 verdicts, worker_threads is forward-risk for Next 15+ but
not the gate today.

**Recommendation: defer to W11.5-E2.5.**  Don't bloat this wave.

5. **Production deploy adapter.**  `@cloudflare/next-on-pages` /

E2's deliverable is bounded: webpack `compiler.run()` completes on a
synthetic input under Nimbus's substrate, and a numeric depth-and-count

of those four remaining Next-blocks have a Nimbus-roadmap path
