## 2. Surprises

### S1 — Miniflare loopback WS-upgrade bug blocks the entire WS-driver test suite locally

The probe driver (`audit/probes/_driver.mjs:55`) opens a WebSocket to `/s/<sid>/ws` to drive shell commands. Against `bun run wrangler dev` in this environment, every WS upgrade fails with:

```
[wrangler:error] SyntaxError: Unexpected end of JSON input
    at JSON.parse (<anonymous>)
    at handlePrettyErrorRequest (.../miniflare/.../index.js:75624:55)
    at #handleLoopbackUpgrade (.../miniflare/.../index.js:86868:48)
[wrangler:error] TypeError: Web Socket request did not return status 101
[wrangler:info] GET /s/<sid>/ws 500 Internal Server Error
```

Miniflare's pretty-error handler tries `req.json()` on what should have been a WS-upgrade request (which has no body), gets an empty buffer, and the `JSON.parse('')` throws. The loopback wrapper then surfaces a 500 and the upgrade never reaches workerd's actual WS handling.

This is a miniflare bug, not a Nimbus bug. It blocks every WS-driven probe in this environment — same root cause as W3 retro S5 (`Local wrangler dev hangs on async unix-commands`, although that was a different surface).

**Mitigation (Phase D pivot):** wrote `audit/probes/w3.5/_local/integration-shim-eval.mjs` — a Node-side harness that materialises the SHIMS string in a `new Function` scope, builds a fixture VFS bundle + dirs map, runs the facet-manager pre-compile loop (with Fix C variants), and exercises `__require()` against three fixtures, one per fix. All three PASS, exercising the same code paths the failing WS probes would have. Documented in `audit/sessions/W3.5-progress.md` Phase D.

The integration harness is only a partial substitute for the full e2e: it doesn't exercise the npm install pipeline, the real workerd CSP eval rejection, or the bundle-cap eviction interaction. Those remain prod-only verifications. But the three fixes are unambiguously validated for the require/resolve flow they were intended to repair.

### S3 — `looksLikeEsm` regex needs careful anchoring

The v0 sketch in W3.5-plan.md §2 had:
```js
const importStmt = /(^|\n)\s*import\s+(['"][^'"]+['"]|[\w*$\{])/;
```

In Phase C the character class `[\w*$\{]` was wrong: `\{` inside a character class is just `{`, but the brace is meaningful only if matched against a literal `{` opener, not against any of the other characters. Fixed to `[\w*$]|\{` (alternation). The 18-case unit test (run during Phase C, then deleted) caught this — initial naive regex incorrectly flagged `const obj = { import: 1 };` as ESM.

### S4 — `EsbuildService` constructor takes a VFS, not unit

I imported `EsbuildService` and assumed `new EsbuildService()`, but the constructor is `constructor(vfs: SqliteVFS)`. Wired the lazy-init path to use `this.vfs as any` (the type is fine; the SqliteVFS reference is what the service needs for cwd resolution during bundling). Existing `nimbus-session.ts` does the same.

### S5 — Backtick character in template-string comments breaks the build

Hit this at `src/node-shims.ts:1848`: my Fix-A docstring used backticks around `` `base` `` to format the variable name. The whole `node-shims.ts` is a template-string from `generateShimsCode()`, and the backtick prematurely terminated the template. TS reported `';' expected` at the column where the spurious template close happened. Fixed by switching to double-quotes: `"base"` instead of `` `base` ``. Phase A plan section also used backticks in a few places — those are in a separate `.md` file, harmless.

### S7 — The `try {} catch {}` swallow at facet-manager.ts:208 was the entire diagnostic-quality bottleneck for ESM regressions

This is documented in W3.5-plan.md §1 Failure 2 but worth re-stating: the silent SyntaxError swallow at the per-file pre-compile step is the SINGLE BIGGEST source of "unsolvable" debug sessions in this codebase. Every package that ships ESM (and there are many) fails identically at request time with the misleading "file was not pre-bundled" — even when the file IS in the bundle, the bundle IS shipped, and the loader IS running. Fix C alone (without Fix B) would have shaved the W3 retro's hours-long jsdom investigation down to seconds.

Forward implication: every silent `try { ... } catch {}` in the facet-manager templates is suspect. Sub-agent review would have flagged this if I'd asked specifically about diagnostic surfaces, but the v0 plan didn't ask. Will look for sibling silent-swallows in W4+ planning.

## 3. Scope deviations

### D1 — Standalone integration harness was bonus

Not in W3.5-plan.md §4 scope — only `audit/probes/w3.5/{functional,regression,e2e}/` were planned. The integration harness `audit/probes/w3.5/_local/integration-shim-eval.mjs` was added in Phase D as a pivot when miniflare blocked the WS-driven probes. ~150 LOC.

It's the strongest local proof of Fix A + B + C in this environment. Worth keeping post-wave because it's wave-agnostic: it exercises any future changes to `__resolveFile`, `__pathIsFile`, the pre-compile loop, or `__loadModule`'s error path.

### D2 — Prod retest deferred (out of session reach)

Plan §6 acceptance gate "All W3 functional+regression probes still PASS" was meant to be empirically verified. With prod pre-W3 (S2) and local wrangler-dev's WS broken (S1), the only way to verify was by-inspection diff review.

The diff is small enough for that to be reliable: Fix A adds one new helper (`__pathIsFile`) and gates one branch of the existing `__resolveFile` loop on it; Fix B is purely additive (new helpers + an optional async pass at end of `buildPrefetchBundle`); Fix C only touches the `try { ... } catch {}` at line 208/386 (now records into a map) and adds a branch to `__loadModule`'s already-existing error-rewriting block. No deletions, no semantic flips on the success paths.

But this is a known weakness of the wave that should be retested in prod after deploy — flagged for the deploy-runner.

### D3 — `EsbuildService` lazy-instantiation is a slight architectural drift

`FacetManager` did not previously depend on `EsbuildService`. Fix B adds an optional dependency: it constructs its own service if `NimbusSession` doesn't share one. This means a fresh facet exec on a brand-new session with no prior `node` invocations now pays a one-time esbuild-wasm init (~10 ms cold) on first exec.

Mitigation: `NimbusSession.ensureFacetManager` shares its session-level `esbuildService` if one exists, so the cost is paid at most once per session even across `node`-shell + `node-script` invocations. The next iteration could pre-warm esbuild on session-start, but the cold-init cost is well below human-perceptible thresholds.

Forward implication for W4: the WASM-swap registry (W6) will probably want a similar shared-service pattern. The lazy-init-with-share approach used here is the cleanest pattern and W6 should adopt it.

### D4 — Fix C is "bonus diagnostic" not "bug fix"

The W3 retro flagged jsdom + fastify as the two acceptance failures; the misleading "file was not pre-bundled" error message wasn't itself called out as a defect. But it directly caused the misdiagnosis of jsdom's failure (sub-agent investigation: "the file IS in the bundle but somehow not pre-compiled" — a hypothesis only reachable because the error message mentioned the bundle).

Fix C went in because it's the lowest-LOC, highest-leverage diagnostic improvement on the wave's hot path. Cost: +27 LOC, no semantic risk. Benefit: every future ESM-shaped regression surfaces immediately at the user terminal with a sane error message.

Bonus scope, no regret.

## 4. Decisions for follow-up waves

### W3.6 candidates

1. **Test-coverage parity for the package.json `"type":"module"` case.**
   The current ESM detector uses regex sniff — works for files with top-level `import`/`export` STATEMENTS. A `.js` file inside a `"type":"module"` package that uses ONLY `module.exports = ...` (rare but legal) wouldn't be detected, would compile cleanly via `new Function`, and load as CJS. That's actually correct — legal CJS. But if the package also has a peer file using `import x from y` IN A DIFFERENT FILE, that one needs the transform. The current implementation handles this case correctly (regex is per-file, not per-package). Worth a regression test that inverts the assumption: a `"type":"module"` package with a CJS-shaped file should still load.

2. **Greedy oversample ESM-priority de-prioritization.**
   `greedyAddMainEntries` (`src/facet-manager.ts:606+`) iterates `meta.module` and `dot.import` candidates and inserts them into a `Set`. Set iteration order is insertion order; `meta.main` goes in first. So `meta.main = cjs, meta.module = esm` results in `[cjs, esm]` and the loop breaks on the first `landed=true` — CJS wins. But for ESM-only packages where `meta.main` is also ESM (astro, @tailwindcss/vite), the greedy adds the ESM file alone — correctly, because there's no CJS alternative. With Fix B that's harmless. Without Fix B it was the bug. Confirmed-correct, no follow-up needed.

3. **Pre-warm esbuild on session start.**
   The first `node` exec on a fresh session pays a ~10 ms wasm-init cost. Pre-warming in `NimbusSession.ensureSqliteFs` or similar would amortise the cost behind the existing session-startup banner. Low priority — the cost is human-imperceptible.

4. **Surface unresolved bare-import warnings from `pre-bundle-facet`.**
   `pre-bundle-facet.ts:558` already pushes warnings into `result.warnings`. The supervisor receives them but doesn't surface to the user terminal — same diagnostic-blackhole pattern Fix C addressed for the runtime path. Worth considering as a sibling fix.

### CT2 (platform-gated tracking) item

- Miniflare's loopback WS-upgrade bug. If a future workerd / wrangler / miniflare update fixes the empty-body request to `handlePrettyErrorRequest`, the entire local-wrangler-dev probe driver will work again. Worth re-testing on every wrangler bump.

### W4 implication

The two-fix-with-diagnostic-surface pattern from this wave (Fix A and Fix B targeted real bugs; Fix C made future regressions cheap to diagnose) is a good precedent. W4+ planning should ask "what's the silent-swallow / misleading-error class for this code path?" alongside the bug-fix question.

### W6 implication

W6 will add WASM-swap registry + REJECT_INSTALL UX. Adopt the same lazy-init-with-share pattern from `FacetManager.setEsbuildService` for any new shared services. Single-source-of-truth contract is more important than performance optimization here — duplicate inits are obviously wrong, but two services with subtly different config drift silently and surface as acceptance gaps several waves later.
